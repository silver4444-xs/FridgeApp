## 一、60+ 组同义词手动整理还是自动挖掘？怎么保证覆盖完整？

### 整理方式

60+组同义词是**手动整理**的。之所以没有采用自动挖掘方案，是基于对烹饪领域特点的判断：食材同义词具有很强的领域封闭性和确定性——比如"番茄=西红柿""马铃薯=土豆=洋芋"——这些映射关系是有限且明确的，用规则字典反而比统计方法更可控，不会出现"土豆"被错误关联到"土豆淀粉"这类过度泛化的问题。

具体来说，从三个来源进行整理：一是菜谱数据集中实际出现的不同叫法，比如323道菜谱里同一食材的不同表达；二是中文烹饪领域的常见别名和方言叫法，比如"洋芋""青瓜"这类地区性表达；三是日常口语中用户可能使用的简略说法，比如"鸡"指代"鸡肉"、"蛋"指代"鸡蛋"。整理完成后做了对称性和无自引用校验，确保每个映射对都是双向可查的。

### 三层覆盖机制

同义词不是只在查询时做一次展开，而是在整个检索链路中做了三层覆盖：

**第一层，索引构建时展开。** 倒排索引在构建阶段就对食谱中的每个食材做了同义词扩展——菜谱里写的是"番茄"，会同时在"西红柿"这个索引键下也插入该菜谱。这样无论用户说哪个词，都能命中。

**第二层，搜索时展开。** 用户的查询词和冰箱里的食材名都会经过归一化（去修饰前缀、去数量单位、小写化）后再做同义词扩展，保证即使用户输入"两个新鲜西红柿"，也能被归一化到"西红柿"再进行匹配。

**第三层，匹配兜底。** 前两层都没命中时，还有一个子串匹配作为最后兜底——比如"鸡胸肉"虽然已有同义词映射到"鸡胸"，但如果出现了一个未覆盖的变体叫法，子串包含关系仍有可能兜住。

### 覆盖完整性问题

手动字典无法做到100%覆盖，但采取了多项措施来最大化覆盖率：

1. **以菜谱数据集为基准验证**——用323道菜谱中实际出现的所有食材名做了一次遍历检查，确保高频食材没有遗漏。
2. **单元测试做硬约束**——所有同义词映射必须满足双向对称性（A→B 则 B→A）和无自引用约束，这是自动化校验的。
3. **子串兜底降低漏召回风险**——前两层同义词匹配没命中时，子串包含关系作为第三道防线。

如果未来要进一步提升，会考虑引入词向量相似度作为第四层软匹配——用预训练的中文词向量对未命中词和索引键计算余弦相似度，超过阈值则视为匹配。但对于当前323道菜谱、60+组高频食材的场景，手动字典配合三层覆盖已经能覆盖绝大部分真实用户的查询需求。

## 二、BM25 参数（k1, b）怎么调的？

### 实际做法

**没有对 BM25 的 k1 和 b 参数做独立调参**，使用的是 rank_bm25 库的默认值（k1=1.5, b=0.75）。但这背后是一个有意识的工程决策，而不是忽略。

### 为什么默认参数在本项目中是可接受的

BM25 在检索架构中的定位是关键。设计的不是一个单一的 BM25 检索器，而是一个**三路混合检索系统**：

- **图索引路径**（实体级 + 主题级双层检索，权重 0.2）
- **向量检索路径**（Milvus 语义检索，权重 0.5）
- **BM25 关键词路径**（权重 0.3）

BM25 在这套架构里扮演的是**稀疏检索兜底**角色——负责捕捉向量检索可能漏掉的精确关键词匹配，比如用户明确提到"糖醋排骨"这类菜名时，BM25 能精准命中。

更关键的一点是，在最终融合阶段，BM25 结果不是用原始 BM25 分数参与加权的，而是用的**基于排名的倒数分**：`1/(1+rank)`。这意味着 BM25 参数的调整只会改变同一路径内部结果的排序，不会影响分数绝对值跨路径的可比性。而在这套三路融合框架中，向量路径占一半权重，BM25 路径的精细化调参对最终排序的影响会因权重而被稀释。

### 真正的调参重心

调参的精力放在了**影响面更大的融合层**上：

1. **三路权重分配**（0.2/0.5/0.3）——基于 Ragas 评估数据集上反复对比得出。向量检索权重最高（0.5），因为烹饪领域的语义匹配场景远多于精确关键词匹配。

2. **文档类型补偿机制**——图索引路径天然偏向菜谱实体（因为有实体名精确匹配），导致烹饪知识类文档在混合检索中容易被压制。设计了一个条件补偿策略：当图索引路径的最高匹配分低于 0.8 时（说明用户问题偏通用烹饪知识而非具体菜谱），对烹饪知识类文档乘以 1.4 的分数补偿；当最高分超过 0.8 时（说明明确匹配到了菜谱），不做补偿。

3. **评估驱动的迭代方法**——构建了一套基于 Ragas 的五指标评测体系（上下文精确率、召回率、忠实度、相关性、正确性），如果未来需要对 BM25 参数做独立调优，可以直接在这个评测框架上做网格搜索。

### 如果要调

k1 控制词频饱和度，b 控制文档长度归一化强度。烹饪场景的特殊性在于：

- **文档长度差异大**——菜谱文档短的几十字（食材列表），长的上千字（含烹饪步骤、营养信息）。b 调低（接近 0）可以降低长度差异对评分的影响，对大文档更公平。
- **词频饱和度**——食材名在菜谱中出现次数少，一般就 1-2 次，k1 的影响相对有限。

调参方法上，在 Ragas 评测数据集上做网格搜索（k1 ∈ [0.5, 2.5], b ∈ [0.3, 1.0]），以检索召回率和精确率的加权调和平均作为目标函数，选出最优组合。

## 三、搜索展开后同义词太多导致召回过多，怎么控制？

系统采用的是**"宽松召回 + 多层后置控制"**的策略。同义词展开本身不做精确度限制，而是在展开之后的排序、截断和融合环节层层收紧。具体有四层控制机制：

### 第一层：字典本身的封闭性控制

60+组同义词是**封闭的手工字典**，不会自动外扩。自动挖掘方案容易引入噪声——比如用词向量发现"土豆"和"土豆淀粉"相似度很高，自动加入后反而会污染召回。手工字典每条都经过验证，保证了扩展的领域纯度。这从根本上防止了"雪崩式展开"。

### 第二层：匹配数作为天然精确度信号

同义词展开后，每个候选菜谱与冰箱食材集做交集，得到一个**匹配食材数量**。这里有一个关键观察：同义词带来的"假阳性"命中，通常只会贡献 1-2 个匹配（比如冰箱有"酱油"，同义词展开出"生抽""老抽"，恰好某菜谱也有"老抽"），而真正相关的菜谱往往有 4-6 个食材同时命中。用匹配数量排序后，低质量的同义词浮票自然沉到底部。

### 第三层：硬截断

排序后直接按 `limit` 参数（默认 5）截断。这是一个看似简单但有效的最后防线——无论同义词展开导致候选集多大，最终返回给用户的结果不会超过这个数量。

### 第四层：在 RAG 路径中的多路融合稀释

在 GraphRAG 混合检索路径中，BM25 关键词检索只占总权重 30%。即使同义词展开让 BM25 路径返回了偏多的候选，向量检索路径（50%）和图索引路径（20%）会用语义相关性对它们重新打分。三路分数加权合并 + 按节点去重后，仅靠同义词关键词匹配上但语义不相关的文档会被向量路径的低分拉下来。

### 进一步优化的思路

当前方案在 323 道菜谱规模下完全够用，但如果扩展到万级菜谱，会做两件事：

1. **同义词分级权重**——核心同义词（番茄=西红柿，精确等价）权重 1.0，宽松同义词（牛肉→牛腩/牛排，其实是子类关系而非等价）权重降到 0.6-0.7。展开时"牛肉"可以命中"牛腩"的菜谱，但贡献的匹配分打折。

2. **匹配率代替匹配数**——用 `匹配食材数 / 菜谱总食材数` 代替绝对匹配数。冰箱里碰巧有一个"糖"的同义词命中了某道复杂菜谱，但该菜谱需要 15 种食材，匹配率只有 1/15，就不会排在前面。本质上是在召回分数里引入精度的维度。

这套"召回层宽松、排序层收紧"的设计思路，是信息检索里的经典范式——召回阶段追求高覆盖，排序阶段再精细区分。

## 四、同义词映射遇到多义词怎么办？比如"包菜"在不同地区可能指不同东西？

### 当前系统的处理方式

**当前系统没有专门的多义词消歧机制。** 这不是疏漏，而是基于两个现实判断做出的工程取舍。

构建同义词字典时，刻意避开了已知的多义词汇。60多组映射集中在烹饪高频食材上，选择的都是**在烹饪语境下单义性很强的词**——"番茄=西红柿""土豆=马铃薯"这样的映射，在厨房场景下没有歧义。

对于一些已有微妙差异的词，采用的是**烹饪场景下的实用等价**而非语言学意义上的同义。比如"酱油→生抽/老抽"，严格来说生抽和老抽是不同的调味品，但对于"冰箱里有没有酱油类调料能用来做这道菜"这个场景，把它们归为一组是合理的——用户不会因为菜谱要"老抽"而冰箱只有"生抽"就放弃这道菜。

这类实用等价映射的底线是：**烹饪上可互相替代，且不会引入安全或口味风险**。反过来，"包菜"没有被纳入字典——正是因为它在不同地区的指代差异太大（卷心菜/生菜/大白菜），且这些指代对象在烹饪上**不可互相替代**，强行映射反而会误导用户。

### 多义词问题的本质

同义词（synonymy）和多义词（polysemy）是检索里的两个不同问题：

- **同义词：多个词指向同一个概念** → 扩展召回，方向是"收拢"
- **多义词：同一个词指向多个不同概念** → 需要区分，方向是"发散"

朴素字典天然适合解决同义词问题，但碰上多义词会适得其反——展开越多，噪声越大。

### 如果要解决，设计思路

当前 323 道菜谱 + 60+ 组食材的规模，多义词还不是瓶颈。但如果要解决，会设计一个**基于上下文的两阶段消歧方案**：

**第一阶段：查询侧消歧。** 用户说"包菜"时，不要立即展开，而是结合冰箱上下文做推断。如果冰箱里已经有"卷心菜"，那"包菜"大概率就是指卷心菜——用一个简单的条件概率即可：P(卷心菜|冰箱已有卷心菜) 远高于 P(生菜|冰箱已有卷心菜)。这是利用**用户当前环境的共现信息**来做消歧，不需要训练模型。

**第二阶段：候选侧排序。** 如果冰箱里没有明确的消歧信号，那就把"包菜"的所有可能指代（卷心菜/生菜/大白菜）都展开去检索，但在排序阶段做惩罚——每个"包菜"候选只能贡献 1/N 的匹配分（N = 可能的指代数），这样多义菜谱不会因为语义模糊而获得不公平的高分。本质上是用**排序阶段的降权**对冲召回阶段的不确定性。

**终极方案**：如果数据规模足够，可以用菜谱中该食材的共现食材来构建词义表征——"包菜"和"干辣椒""花椒"共现时更可能是卷心菜（手撕包菜的做法），和"沙拉酱""橄榄油"共现时更可能是生菜。这本质上是用**分布语义**来区分词义，类似 Word Sense Disambiguation 的经典思路，但不需要外部知识库，直接在菜谱语料上就能做。

### 总结

当前字典设计的原则是"宁缺毋滥"——烹饪上可替换的才建映射，不可替换的多义词宁可留空也不强行关联。这种保守策略在 323 道菜谱规模下保证了精确率，舍弃的只是少量边缘召回。

## 五、三层覆盖中哪一层对命中率提升最大？做过消融实验吗？

### 坦诚回答

**没有做过形式化的消融实验。** 这是一个真实存在的不足。但可以基于系统设计原理给出分析，并说明如果要做的实验方案。

### 基于设计的定性判断

三层覆盖的定位本质不同，它们解决的是不同粒度的匹配失败：

**第一层（索引构建时展开）是"基础设施"。** 没有它，后续所有搜索都无法利用同义关系。但这一层的效果依赖搜索时冰箱侧也被展开（第二层），否则会出现方向性缺失——比如菜谱写"番茄"被展开索引到"西红柿"键下，但冰箱侧的"番茄"如果没有第二层展开，搜索集里就没有"西红柿"这个键，根本不会去查那个索引项。所以**第一层和第二层是配套的，单独一层只解决了一半问题**。

**第二层（搜索时展开）是"放大器"。** 它把冰箱食材集从精确名扩展到所有同义名，扩大了搜索键集合。从匹配机制看，模糊查找本质上是在索引中逐个查找搜索键，键越多命中越多。第一层铺好了路（索引里有同义键），第二层派出了更多车（搜索集有更多键），两者协同才产生召回提升。

**第三层（子串兜底）是"安全网"。** 它覆盖的是前两层都搞不定的情况——食材变体不在60组同义词字典里。比如"去皮鸡胸肉"经过归一化去掉"去皮"后还剩"鸡胸肉"，而同义词字典里恰好有"鸡胸肉→鸡胸"，所以会被第二层兜住；但如果是一个完全未知的变体（比如"清远鸡"），归一化后字典里没有，那就靠子串匹配碰运气了。

定性判断是：**第一层 + 第二层的协同贡献最大（覆盖已知同义词），第三层贡献较小但覆盖的是"长尾未知变体"这个不同的问题。** 而归一化（去修饰前缀、去数量单位）是所有这些之前的前置步骤，没有它，后续三层都无从谈起——这一点在单元测试中有单独的覆盖验证。

### 如果要补消融实验

实验设计上，在 Ragas 评测框架上做**逐层剥离**：

| 配置 | Layer 1 索引展开 | Layer 2 搜索展开 | Layer 3 子串 | 归一化 |
|------|:---:|:---:|:---:|:---:|
| 基线 | ✓ | ✓ | ✓ | ✓ |
| 剥离子串 | ✓ | ✓ | ✗ | ✓ |
| 剥离搜索展开 | ✓ | ✗ | ✗ | ✓ |
| 剥离索引展开 | ✗ | ✗ | ✗ | ✓ |
| 剥离归一化 | ✗ | ✗ | ✗ | ✗ |

用 323 道菜谱的全部食材做查询集（每道菜谱随机抽 1 个食材作为查询词，共 323 条查询），观察每种配置下**召回率**（该菜谱是否出现在结果中）和**平均候选集大小**（反映精确度）。倒排索引的模糊查找是确定性的纯规则方法，实验结果没有方差，不需要多次重复——这一点比需要做显著性检验的向量检索实验简单得多。

已有完整的 Ragas 评测基础设施（50条问答、5指标），这个消融实验可以直接嵌入现有测试框架中，作为检索模块的单元级评测。

### 反思

消融实验的缺失是一个值得改进的地方。当前系统的评测重心在 RAG 端到端生成质量（忠实度、相关性等），对检索模块内部各子策略的贡献缺乏精细度量。这导致关于"哪一层贡献最大"的判断只能靠设计直觉而非数据。接下来如果要写论文或答辩，补上这组实验会是一个有力的补充。

## 六、如果用 Embedding 直接做语义匹配，还需要同义词映射吗？什么场景下规则方法比语义方法更好？

### 有了 Embedding 还需要同义词映射吗？

**大多数情况下不需要，但少数关键场景下仍然需要。** Embedding 解决的是"语义相近"的问题，但不能保证"语义等价"。这两个概念的差距，就是规则方法仍然有价值的空间。

**第一个原因：训练分布中的对比共现导致语义距离拉大。** 比如"生抽"和"老抽"在烹饪语境下是可互相替代的调味品，但在美食文章、电商评论中，这两个词常常以对比或并列的方式出现（"用生抽还是老抽？""生抽提鲜，老抽上色"），导致它们的 Embedding 向量反而被推远了。训练分布中的对比共现导致语义距离拉大，是同义词规则不可替代的第一个原因。

**第二个原因：方言词和口语变体的稀疏性。** 像"洋芋""青瓜"这类地区性词汇，在通用预训练语料中出现频率极低，Embedding 模型对它们的表征可能不可靠。规则字典里一条映射就能解决的事，用语义匹配反而有随机性。

**第三个原因：精确度边界。** 语义匹配天然带有模糊性，阈值设高了漏召回，设低了误召回。而规则匹配是确定性的——要么命中，要么不命中，没有中间地带。在某些对精确度要求苛刻的场景下（比如用户明确说"我忌口牛奶"，你不能把"酸奶"的菜谱推给他），确定性的等价判断比向量相似度更合适。

### 什么场景下规则优于语义？

结合项目实践经验，总结四种场景：

**场景一：冷启动与小语料。** Embedding 模型的领域适应性依赖微调或足够丰富的领域预训练语料。当菜谱只有 323 道、食材词表只有几百个时，建一个 60 组的同义词字典一天就能完成，验证和调试一目了然。而 Embedding 在这个规模下反而可能因为缺乏足够上下文而表征不稳定。规则方法不依赖数据规模，即插即用。

**场景二：确定性等价，而非相似性。** 同义词映射表达的是"番茄 = 西红柿"这样的等价关系，而 Embedding 表达的是连续语义空间中的距离关系——"番茄"离"西红柿"很近（0.95），但离"圣女果"可能也不远（0.82），再远一点可能是"番茄酱"（0.70）。如果想精确地只召回等价食材而非相关食材，排序阶段靠阈值很难一刀切。规则字典的二元判断（是/否）天然适合等价关系建模。

**场景三：可解释性与调试。** 当用户反馈"为什么没搜到糖醋排骨？我冰箱明明有糖"，规则路径的排查是透明的——可以追溯：用户输入"糖"→归一化→同义词展开为["白糖","细砂糖","冰糖"]→查倒排索引→命中。而 Embedding 路径只能告诉你"向量相似度 0.73，没达到某个阈值"，无法解释为什么是这个分数。在需要向用户解释推荐理由或内部调试检索失败原因时，规则方法有天然优势。

**场景四：延迟和资源。** 规则字典的展开是内存查表，O(1)，零模型推理开销。Embedding 需要把查询编码成向量，再在向量数据库里做近邻搜索。在物联网边缘设备（比如冰箱本身的 RK3588 芯片）上，模型推理的延迟和功耗可能是不可接受的。规则方法可以完全在 CPU 上毫秒级完成。在当前项目中，同义词展开和倒排索引查找的总耗时在 1ms 以内，而一次 Milvus 向量检索需要 20-50ms。

### 两者不是替代，是互补

这也是设计混合检索时同时保留两条路径的原因——规则路径保证已知等价关系的**确定性召回**，语义路径覆盖未知表达的**泛化召回**。在最终融合阶段，两者各自贡献自己的优势，加权合并。这种"规则兜底 + 语义扩展"的架构，本质上是用确定性方法守住精确率的下限，用统计方法拓展召回率的上限。

这个设计哲学其实不限于食材匹配——任何领域封闭、术语可枚举、等价关系明确的小型知识体系，都可以用同样的思路。
