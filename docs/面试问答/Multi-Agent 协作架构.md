# FridgeAI 项目面试问答记录

> 基于 FridgeAI 智能冰箱对话系统项目的技术面试问答整理

---

## 问题 1：为什么用 3 个子 Agent 而不是 1 个 Agent 配 3 套工具？

**核心原因：缩小工具选择空间，降低大模型决策错误率。**

### 1. 工具选择空间的组合爆炸

如果单个 Agent 直接面对全部 8 个工具，每次推理需要从 8 个候选中选择正确的工具组合。当用户说"西红柿可以换什么"，Agent 需要在菜谱推荐工具、食材替换工具、烹饪知识检索工具之间抉择——这三个工具都涉及食材概念，但语义上完全不同。实际测试中发现过"用户问替换，Agent 调了推荐接口"的误路由情况。

拆成 3 个子 Agent 后，每个子 Agent 只暴露 1 到 3 个语义高度内聚的工具：菜谱专家只持有推荐、搜索、详情三个工具；替换专家只有一个替换工具；烹饪专家只有一个知识检索工具。主 Agent 的工具选择空间从 8 缩小到每次只需在 3 个专家间做路由决策，子 Agent 内部进一步缩小到 1 到 3 个。

### 2. 系统提示词的专注度差异

三种场景需要完全不同的输出格式和角色设定。菜谱推荐要求 Markdown 表格格式、最多 5 个推荐、按匹配度排序；食材替换要求 2 到 3 个替代方案并标注口味影响、优先推荐冰箱已有的；烹饪知识要求引用检索结果、最多 300 字、使用标题分隔。如果把这三种行为规范塞进同一个提示词，指令越多模型遵循度越差——这是提示词工程中公认的"指令稀释"问题。独立子 Agent 让各自的提示词短而精准。

### 3. 独立推理参数控制

替换专家使用了温度参数设为 0（确保替代建议的一致性——同一个缺失食材每次给出的替换方案应该稳定），而菜谱推荐允许一定随机性（温度参数 0.1）。单 Agent 无法对不同工具调用场景设置不同的推理参数。

### 4. 路由决策下沉

主 Agent 的职责简化为"判断用户意图 → 路由到正确的子 Agent"，不需要自己执行复杂的多步推理。每个子 Agent 被调用时已经是带着明确任务的独立执行单元，主 Agent 拿到结果后直接整合输出。这符合分治思想。

### 代价

多一次 Agent 调用意味着多一次大模型 API 请求，延迟和 Token 消耗会增加。但在本项目中，菜谱推荐本身就需要多轮工具调用，相对额外开销占比不大，而准确性提升是实打实的。项目中配置的 5 层中间件（调用限流、摘要压缩、人工审批、模型重试、工具重试）也部分承担了对这个额外开销的控制。

## 问题 2：人机审批（HITL）在项目演示中真的实现了吗？具体什么触发审批？

**已完整实现，从中断触发到恢复执行的全链路代码可运行。**

### 触发入口

当前唯一触发审批的操作是"保存用户饮食偏好"。当 Agent 决定修改用户的忌口、菜系偏好、人数等设置时，中间件会在工具实际执行前拦截并中断。

### 完整流程

1. **Agent 执行中触发**：用户说"我不吃辣"，Agent 推理后决定调用偏好保存工具写入饮食限制
2. **中间件拦截**：人工审批中间件在工具实际执行前中断，将当前状态（包括对话历史、中断点信息）保存到检查点存储器，然后抛出中断信号
3. **服务端通知客户端**：WebSocket 流式处理模块捕获中断事件，通过 WebSocket 向前端发送中断通知，前端展示审批卡片（含"同意"和"拒绝"按钮）
4. **用户决策**：用户点击同意或拒绝，前端发送恢复消息（含决策类型）
5. **恢复执行**：服务端收到恢复消息后，调用恢复指令从检查点存储器恢复中断前的状态。同意则真正写入偏好，拒绝则跳过该工具调用，Agent 继续执行

### 当前局限

中断只在 WebSocket 流式对话路径中完整生效。如果通过 REST 接口直接调用 Agent，中断虽然触发但恢复逻辑未在 REST 接口中实现。这是为了优先覆盖主流交互路径（移动端 AI 对话界面）的取舍。

代码中预留了未来可扩展的审批点（清空冰箱、删除收藏菜谱等），但目前仅偏好保存一个入口。

## 问题 3：调用限流用什么算法？阈值怎么定？

**算法是简单的计数器对比，不是滑动窗口或令牌桶。**

### 算法细节

中间件维护两个计数器：
- **单次调用计数**：一次对话请求中的大模型调用次数，不跨会话持久化
- **线程级计数**：同一会话线程下所有请求的累计次数，持久化到状态中

每次模型调用前检查计数器是否达到上限，调用后计数器加一。触发后的行为配置为"优雅结束"——直接跳转到 Agent 执行末端，注入一条提示消息告知用户已达调用上限，而不是抛异常导致界面崩溃。

### 阈值 15 的由来

当前只设置了单次调用上限 15，没有设置线程级上限。这个数字不是通过严格数学推导得出，而是基于项目实际交互模式的经验估算：

- 简单对话（"冰箱有什么"）：推理 → 调 1 个工具 → 组织回复，约 2 到 3 次模型调用
- 复杂对话（"推荐菜谱 + 缺食材替换 + 烹饪做法"）：路由到子 Agent → 菜谱专家内多次推理 → 替换专家 → 汇总回复，约 6 到 8 次
- 考虑子 Agent 内部 2 到 3 轮工具调用，加上模型重试最多 3 次，极端情况约 10 到 12 次
- 15 比极端情况多留 3 到 5 次余量，够用但不至于让死循环跑太久

### 改进方向

更严谨的做法应该是基于线上统计数据的 P95/P99 分位数来定阈值，或者用线程级计数做跨轮次总量控制。但在演示阶段，简单计数器加经验阈值已经是务实的防死循环方案。

## 问题 4：如果菜谱专家推荐的菜谱需要替换专家才能找到替代食材，两个 Agent 怎么协作？

**它们不直接协作。当前架构是主 Agent 串行编排模式，不是子 Agent 间互相调用。**

### 实际协作路径

菜谱专家的内部工具只有菜谱推荐、食材搜索、菜谱详情三个，它完全不知道替换专家的存在。同样，替换专家只有一个食材替换工具，也不知道菜谱专家。它们互相不可见。

一个典型的"推荐 + 替换"场景流程：

1. 用户说"能做什么菜？缺的食材能换吗？"
2. 主 Agent 分析意图，先调用菜谱专家
3. 菜谱专家返回结果（含匹配食材和缺少食材标注）
4. 主 Agent 读取输出，发现标注了"缺少食材"，然后决定调用替换专家询问替代方案
5. 替换专家返回替代建议（含口味影响说明）
6. 主 Agent 将两边结果整合成最终回复

### 核心问题

整个过程依赖主 Agent 能否"悟到"该做这个链式调用。主 Agent 的路由规则是"推荐菜 → 菜谱专家""替换 → 替换专家"这样的硬划分，没有"推荐后自动触发替换"的联动指令。如果用户只问"能做什么菜"而不提替换，主 Agent 调完菜谱专家就结束了，不会主动补一刀替换查询。

### 改进方向

更好的做法应该是让菜谱专家的输出里带上缺失食材的适配标记，或让主 Agent 的系统提示词明确加入"检测到缺失食材时主动调用替换专家"的指令。但在演示阶段，用户主动问替换时链路是完整的，所以这个边界在实际使用中不太会被触发。

## 问题 5：五层中间件的执行顺序？重试逻辑会叠加吗？

### 执行顺序

中间件列表的顺序即为管道从外到内的包裹顺序（洋葱模型）：

1. **调用限流中间件**（最外层）— 每次模型调用前检查计数器，调用后递增
2. **摘要压缩中间件** — 检查对话 Token 是否超过 4000，超过则触发摘要
3. **人工审批中间件** — 拦截特定工具调用（偏好保存），触发中断
4. **模型重试中间件** — 包裹底层大模型 API 请求，异常时重试（最多 3 次）
5. **工具重试中间件**（最内层）— 包裹工具函数执行，异常时重试（最多 2 次）

执行流程：请求进入 → ① 限流检查 → ② 摘要检查 → ③ 审批拦截 → ④ 模型重试包裹 → ⑤ 工具重试包裹 → 实际执行 → 响应按反向顺序 ⑤④③②① 回收。

### 重试不会叠加

两个重试中间件作用于不同层次：

- **模型重试**包裹的是大模型 API 的 HTTP 请求。重试时在钩子内部完成，不会重新触发外层中间件的检查钩子。所以调用限流中间件看到的是"1 次模型调用"，无论里面重试了几次。
- **工具重试**包裹的是工具函数的执行。同样在钩子内部完成，不触发额外的模型调用钩子。

**真正的叠加效应在 Agent 循环层面**：工具调用失败（即使工具重试已耗尽）后，Agent 会重新调用大模型分析错误信息并规划下一步——这会触发新一轮的模型调用，从而计入限流计数器。这不是中间件层面的技术叠加，而是 Agent 循环中"失败 → 重新推理 → 再次调用"的业务叠加。

**退避参数各自独立**：模型重试使用 1 秒 → 2 秒 → 4 秒（上限 30 秒）的指数退避；工具重试使用 0.5 秒 → 1 秒（上限 10 秒）。两者各算各的，不会乘法叠加。

## 问题 6：如果 Fridge 要服务百万用户，五层中间件的瓶颈在哪？

**最致命的是共享基础设施瓶颈，而非中间件逻辑本身的性能。**

### 1. 内存级状态存储（最致命）

五层中间件中有三层依赖检查点存储器或键值存储来持久化状态：
- 调用限流的计数器存在会话状态里，依赖检查点持久化
- 人工审批的中断状态存在检查点中，恢复时需要精确读出中断位置
- 摘要压缩后的消息替换、用户偏好存在键值存储里

当前代码使用了进程内内存存储。百万用户场景下：
- 进程重启 → 所有会话状态丢失（包括待审批的中断）
- 水平扩展到多实例 → 用户在实例 A 创建的会话，实例 B 无法找到中断状态
- 内存线性增长 → 每个活跃用户一份会话状态，百万用户的内存需求不可行

代码中已标注了技术债：生产环境需替换为关系型数据库支持的持久化存储。这是上线前必须解决的问题。

### 2. 摘要中间件：大模型成本的隐性放大

摘要触发阈值是 4000 Token。长对话中每超过一次阈值就额外调用一次大模型做摘要。百万用户场景下，假设平均每个活跃用户一天产生 3 轮超过阈值的对话，每天多出数百万次额外的大模型 API 调用。摘要提示词本身还包含完整对话历史作为输入，Token 消耗不小。这是成本层面的瓶颈，而非纯技术层面，但对商业模式有直接影响。

### 3. 重试中间件：API 压力放大镜

模型重试（最多 3 次）和工具重试（最多 2 次）在正常情况下不触发。但大模型 API 出现间歇性抖动时，重试会把瞬时请求量放大 3 到 4 倍。工具重试中配置的工具涉及大模型调用（子 Agent 本身就是独立 Agent 实例，内部又有自己的模型调用），工具重试也会链式触发新的模型调用。百万用户时，重试放大的 API 压力可能触发新一轮限流形成正反馈。

### 4. 人工审批：长生命周期会话管理

人工审批不消耗算力，但它让一个对话会话变成长生命周期（等待用户审批可能几秒到几分钟不等）。百万用户意味着可能有数十万待处理的中断状态需要持久化和管理。审批超时后的清理策略目前未体现。

### 优先级

从内存存储迁移到持久化存储是最紧迫的（否则无法多实例部署），其次是摘要触发的成本控制和重试退避策略的调优。其他方面在这个量级下并非硬阻塞。

## 问题 7：和 LangGraph 检查点 + 中断机制比，五层中间件的差异？

**核心差异在于控制粒度和在哪个抽象层解决问题。**

### 当前架构

状态图只有一个"推荐"节点（入口 → 推荐 → 出口），Agent 和五层中间件是这个节点内部的执行体。状态图负责对话历史的多轮持久化（通过检查点），中间件负责 Agent 执行时的运行时治理。

### 1. 中断粒度：节点级 vs 工具级

LangGraph 原生中断的逻辑是"在某个图节点执行之前/之后暂停"，控制粒度是图节点。如果要在"保存偏好"前中断，需要把偏好保存拆成独立的图节点，然后配置在该节点前中断。这要求把 Agent 的动态工具调用循环拆解为静态的图拓扑。

中间件的人工审批直接在 Agent 内部的工具调用层面拦截，不需要重构图结构，不需要预设"哪些工具可能被调用"的静态拓扑。Agent 自己动态决定调用偏好保存工具，中间件在那一刻拦截——图结构完全不变。

### 2. 检查点语义：超级步 vs 步内

LangGraph 的检查点在每个超级步（图节点完成）后自动存档。当前单节点架构下，所有工具调用和中间件逻辑都发生在一个图节点的一次执行内——检查点只能看到节点开始前和结束后的状态。

中间件的计数器（如模型调用次数）和中断状态存在会话状态的私有字段里，靠节点执行结束时一并写入检查点。这意味着如果节点执行到一半崩溃，中间件层的状态可能丢失；而 LangGraph 原生中断的节点边界处，状态是完整的。

### 3. 可观测性：追踪里的一个黑盒

当前图只有一个节点，链路追踪工具里就是一个跨度。中间件的摘要触发、限流触发、重试都在这个跨度内部，从外部看是整体不透明的。如果用 LangGraph 原生的多节点分解（如分析食材 → 检索菜谱 → 排序推荐 → 生成回答），每个步骤是独立的追踪跨度，哪个环节慢了、哪个环节失败了，直接可定位。

### 4. 中间件独有的能力

调用限流、模型重试、工具重试、摘要压缩这四个中间件在 LangGraph 中没有开箱即用的对应机制。如果用纯 LangGraph：
- 需要在每个调用模型的节点里手写重试逻辑
- 在状态中维护计数器并在适当位置检查
- 在合适位置插入摘要节点

中间件把这些问题提升到了框架层，一次配置全局生效——代价就是图层面的不可见性。

### 5. 两者的协作方式

当前代码中，LangGraph 的检查点和中间件的存储共享同一个实例。人工审批中断触发后的完整链路：
- 中间件写入中断状态 → Agent 会话状态更新 → 图节点结束时检查点持久化
- 下次恢复指令到达时 → 图从检查点恢复 → 中间件从恢复的状态中读取中断位置 → 继续执行

LangGraph 提供持久化运输层，中间件提供细粒度拦截逻辑，各管各的层。

### 一句话总结

LangGraph 管"图怎么走"，中间件管"Agent 在一个节点里怎么跑"。前者静态、可追踪、适合多节点编排；后者动态、减少图复杂度、适合 Agent 自主决策的场景。当前单节点架构下两者是叠加关系，若要升级为多节点流水线架构，调用限流和重试类中间件仍需保留，但人工审批可考虑下沉到图节点中断机制以获取更干净的检查点语义。
