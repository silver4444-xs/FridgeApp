# FridgeAI 技术面试问答记录

> 2026-07-20 | 面试官提问，基于项目实际情况作答

## 一、Neo4j 和 Milvus 怎么部署的？Docker？数据量？

**Neo4j** 通过 Docker Compose 部署 `neo4j:5.26-community` 社区版镜像，分配 512MB 堆内存和 128MB 页缓存，Bolt 协议走 7687 端口，HTTP 管理界面走 7474 端口，均绑定 127.0.0.1。数据和日志挂载到命名卷做持久化，Backend 启动前通过健康检查确保 Neo4j 已就绪。

**Milvus** 采用嵌入式 Lite 模式，通过 `pymilvus` 的 MilvusClient 以内嵌方式与 Backend 同进程运行，数据落盘为本地文件。向量索引用 HNSW（M=16, efConstruction=200），余弦相似度度量。配置上保留了 `standalone` 模式的切换能力，通过环境变量即可切换。Lite 模式的选择原因是当前约 800 条向量，Lite 完全可承载，同时省掉一个独立容器的内存开销。

**数据规模**：Neo4j 含 323 道菜谱、500+ 种食材、3000+ 烹饪步骤节点，关系类型包括 REQUIRES、CONTAINS_STEP、BELONGS_TO_CATEGORY。Milvus 中约 800 条文本块 embedding，向量维度 512（BGE-small-zh-v1.5）。

**选型考量**：整机 2C4G，Neo4j 512MB + Backend（含 Milvus Lite + embedding 模型）约 800MB + Nginx 50MB，刚好跑在 2GB 内存线内。后续可通过 Milvus Lite → Cluster、Neo4j Community → Enterprise/Causal Cluster 做水平扩展。

## 二、三种路由怎么判断该走哪个？意图分类器还是规则？

主体用 **LLM + 结构化输出**做意图分类，辅以规则降级兜底。

分类器将查询分析定义为一个 Pydantic 模型，包含五个维度的特征：查询复杂度（0-1）、关系密集度（0-1）、是否需要推理、实体数量、推荐策略与置信度。通过 prompt 让 LLM 对这五个维度打分，用 `with_structured_output`（function calling 模式）直接拿到结构化分析结果。

三种策略的分流逻辑：
- **Hybrid**：简单查找，如"冰箱里有什么""鱼香肉丝怎么做"，走 BM25 关键词 + 向量语义混合检索，不启动图遍历
- **Graph RAG**：复杂推理，如"番茄和什么搭配最有营养""为什么这个菜用蒸不用炒"，走 Neo4j 多跳遍历
- **Combined**：两者都沾边，图 RAG 和传统检索各取 top-k，Round-robin 交替融合去重

降级设计：LLM 调用失败时自动降级到基于关键词的规则判断（命中"为什么""关系""影响"等词汇倾向图 RAG），降级置信度标 0.6；检索过程抛异常时回退到传统混合检索，保证系统不会因图查询失败而白屏。

**为什么用 LLM 而不是纯规则**：中文烹饪查询表达灵活（如"番茄炒蛋和西红柿炒鸡蛋有啥不一样"），关键词规则难以精准区分菜品对比和食材替代；规则无法感知隐含推理需求（如"孕妇能吃这个菜吗"表面是查找，实际需要食材-营养-禁忌的多跳推理链）。分类任务本身 token 消耗很小（一次几十个 token），不会成为性能瓶颈。

## 三、Round-robin 为什么不用 RRF（倒数排名融合）？

**核心矛盾是两个检索器的分数不可通约。** 图检索的相关性来自 Neo4j 的图结构连通性——节点之间的路径数、跳数和关系边权重，衡量的是结构关联度。混合检索（BM25 + Milvus）的相关性来自 BM25 词频分数和余弦相似度，衡量的是语义相关度。RRF 用 `1/(k+rank)` 统一映射时，隐含假设两个检索器的"排名质量"是等价的，但图检索的 rank=1 可能只是因为只有 1 个邻居节点，而非高度相关——这个"第 1 名"的含金量与从 800 条中筛出的向量检索第 1 名完全不同。

**结果集规模不对称**：图检索常常只返回 1-3 个有效结果，而向量检索可靠返回 top-k。RRF 下，图检索 rank=1 得分约 0.016，向量检索 rank=5 得分约 0.015——两个不同语义空间的结果分数趋同，融合退化成了随机排序。

**Round-robin 在当前场景的合理性**：(1) 图结果优先穿插，编码了"走到 Combined 的查询有图推理需求，图结果召回精度更高"的先验；(2) 交替排列保证来源多样性，避免一种来源完全挤掉另一种。

**什么时候该用 RRF**：当检索器数量 ≥ 3、每个检索器都能产出稳定有区分度的排名、候选集规模百级以上时，RRF 优势才真正发挥。当前两个检索器、top_k=5 的小规模下，Round-robin 配合图优先策略更有针对性。

## 四、KG 里有"番茄炒蛋"和"西红柿炒鸡蛋"两个节点，多跳查询会丢结果吗？怎么处理？

**如果 Neo4j 里确实有两个独立节点，单靠图遍历是会丢结果的。** 图遍历基于节点 ID 精确匹配，两个菜谱节点之间没有别名关系边，从"番茄"食材出发的 REQUIRES 边只会指向其中一个。

当前用三层防御补偿：

**第一层，图索引的四级渐进匹配。** 实体检索不直接查 Neo4j，而是通过内存 KV 索引——查询时走"精确匹配 → 同义词扩展匹配 → 子串匹配 → difflib 模糊匹配（阈值 0.6）"。查"番茄炒蛋"时精确匹配命不中"西红柿炒鸡蛋"，但子串"炒蛋"可能命中，模糊匹配也能捞回相似度高于 0.6 的菜名。这相当于在图遍历入口处就做了实体消歧。

**第二层，图内基于名称的去重。** 索引构建完成后，将完全同名的节点合并——保留第一个，把其他节点的内容追加到主节点，删除冗余节点和关系边。这解决了"同一个名字存在多份"的问题，但不解决异名同义。

**第三层，食材侧同义词映射。** 60+ 组食材同义词表在倒排索引构建时展开——构建"番茄"的倒排项时，将所有含"西红柿"的菜谱也关联进同一索引条目。即使 KG 里菜谱节点名用了"西红柿炒鸡蛋"，向量和关键词层面的检索也能通过食材路径捞出它。

**根本解法**：在 Neo4j 中为同义菜谱建立 SAME_AS 或 ALIAS_OF 关系边，让图遍历通过别名边自动跳转；或在数据清洗阶段用实体链接模型做全局菜谱名称归一化。当前未做的原因是 323 道菜谱的规模下同名/同义比例低、实际影响有限，且引入实体链接会显著加重数据管线。

## 五、Combined 路由下，Graph 返回 10 条、Milvus 返回 20 条但完全不重叠，怎么融合？

**零重叠场景下，Round-robin 就是盲插——没有任何质量信号指导谁该排在前面。**

零重叠的两种可能：(1) 两个引擎各自捕捉了查询的不同侧面，两者互补都该保留；(2) 其中一个跑偏了，返回了噪音，Round-robin 会让噪音均匀穿插在有效结果中干扰 LLM 生成。

**正确解法：加一层重排序**——融合后对所有 30 条候选做独立相关性打分，再排序列取 top-k。三个可选方案：

| 方案 | 做法 | 优点 | 代价 |
|------|------|------|------|
| Cross-encoder 重排 | BGE-Reranker 对 (query, doc) 对打分 | 精度最高，成熟方案 | 增加约 100-200ms 延迟 |
| LLM 列表重排 | 把所有候选喂给 LLM 选最相关 k 个 | 捕捉复杂语义 | token 消耗大，延迟高 |
| 分数归一化重排 | 各自分数 min-max 归一化到 [0,1] 排序 | 零额外延迟 | 跨检索器分数本质上不可比 |

最合理选择是 **Cross-encoder 重排**：BGE-Reranker 模型小（几百 MB），可与 embedding 模型共享推理基础设施，对中文烹饪领域泛化能力足够。

实际上系统的混合检索（三路融合：BM25×0.3 + 向量×0.5 + 图×0.2）已经做了一套加权打分机制，证明了基于分数的重排在架构上可行。Combined 路由没有复用这套机制而是退化成了 Round-robin，这是路由层和融合层之间缺少统一重排抽象的问题。

**补充视角**：如果图检索和向量检索频繁零重叠，可能反映索引层面两个检索器覆盖的文档空间几乎没有交集——不是融合策略的问题，而是数据一致性问题。当前系统中 Neo4j 和 Milvus 的数据源是同一批语料，零重叠更可能是特定查询下的个案，重排序足以解决。

## 六、菜谱从 323 扩展到 323 万，Neo4j 和 Milvus 各自瓶颈在哪？

### Neo4j 瓶颈

**内存是第一道墙**：323 万菜谱对应节点总量约 3000-5000 万，关系边过亿。512MB 堆+128MB 页缓存在亿级关系面前等于零，每次图遍历触发磁盘随机读。Community 版不支持分片和集群，所有数据必须塞进单机。

**多跳遍历扇出失控**：从"猪肉"出发 1 跳命中几十万道菜，2 跳展开步骤节点，中间结果百万级膨胀。LIMIT 只对最终结果截断，中间遍历该膨胀还是膨胀，内存和 CPU 先扛不住。需要图投影+预计算子图，或 GraphSAGE/Node2Vec 等图嵌入压缩。

**当前硬编码 LIMIT 1000 作废**：实体索引构建 `MATCH (n) ... LIMIT 1000` 在百万级节点下只摸到冰山一角。

### Milvus 瓶颈

**Lite 模式直接报废**：8M × 512 维 float32 原始数据 16GB，加 HNSW 图结构（M=16 时每节点 32 条出边），总内存 25-30GB，单进程扛不住。必须切换到 Milvus Standalone 或 Cluster。

**HNSW 参数矛盾**：efConstruction=200 在 8M 条时构建需数小时。搜索时 ef 越大召回越高但越慢，ef=64 约 50-100ms，ef=128 翻倍。需要做召回率-延迟 trade-off 调参，甚至引入 IVF+PQ 量化索引。

**分区策略必须引入**：按菜系、品类、难度建立分区键，查询时先分区裁剪掉 90%+ 候选空间。

### 连锁反应

- 内存 KV 索引和倒排索引从 dict 膨胀到几 GB → 需 Redis/RocksDB
- Embedding 全量构建变成 GPU 小时级任务 → 需异步消息队列流水线

| 维度 | Neo4j | Milvus |
|------|-------|--------|
| 架构 | Community 单机 → Enterprise/集群 | Lite 嵌入 → Standalone/Cluster |
| 内存 | 512MB 面对亿级关系 | 25-30GB，需分布式 |
| 计算 | 多跳遍历扇出 O(n²) 风险 | HNSW 构建小时级 |
| 索引 | LIMIT 1000 截断 | 需分区键+量化索引 |
| 附属 | 内存索引 → Redis/RocksDB | Embedding → 异步队列 |

**Neo4j 瓶颈在图拓扑的规模爆炸（关系边比节点长得快得多），Milvus 瓶颈在向量密度的线性增长。** 两者之间还有数据一致性问题——百万级下的双写和增量同步本身就是分布式一致性的难题。

## 七、和微软 GraphRAG（全局/局部社区摘要）方案比，差异在哪？

微软 GraphRAG 核心思路：Leiden 社区检测 → LLM 预生成社区摘要，把图结构"压平"成自然语言段落，查询时读摘要而非遍历图。

五个维度的关键差异：

**一、图的用法：预计算摘要 vs 在线遍历。** 微软把图当作中间产物，最终交付社区摘要文本；在线检索时读摘要。我们把图当作一级检索源，不做离线摘要，查询时在线执行图遍历+多路融合。根本区别：微软计算量前置（昂贵摘要生成）换低延迟查询；我们计算量分散在线，换索引更新的即时性。

**二、数据性质决定路线。** 微软定位非结构化文档语料（PDF、邮件），没有现成关联结构，必须靠 LLM 抽实体和关系才能成图，社区检测弥补"非结构化数据缺乏全局组织"的缺陷。我们的数据从源头就是结构化的——菜名、食材列表、步骤序列、分类标签，实体关系直接映射，Leiden 社区检测边际价值低。

**三、查询模式不同。** 微软强于全局性问题（"10000 份报告的核心主题是什么"），全局检索 mode 天然为归纳设计。我们需要精确检索和局部推理（"冰箱里有番茄鸡蛋能做什么""红烧肉炖多久"），不需要全局摘要，社区摘要反而模糊掉单道菜的具体信息。

**四、成本模型截然不同。** 微软离线成本极高——每次索引更新对所有社区重新生成摘要，数万到数十万 token。扩展到 323 万菜谱，社区数量膨胀到数千数万，成本不可承受。我们离线只有 embedding 计算和索引构建，不涉及 LLM 调用；新增一道菜谱只需 embedding 生成+图节点插入即可上线。

**五、混合架构的可能性。** 如果后续扩展"菜谱文化全局分析"（如"川菜和苏菜的烹饪哲学核心差异""中国家庭最常用 10 种调味组合"），微软路线有明显优势。更优方案是混合架构：日常精确查询走在线检索，全局分析查询走预计算社区摘要，两套索引共存，由路由器按查询类型分流。

| 维度 | 微软 GraphRAG | FridgeAI 当前方案 |
|------|-------------|-----------------|
| 图角色 | 中间产物 → 压成摘要 | 一级检索源，实时遍历 |
| 计算重心 | 离线重（社区摘要） | 在线重（图遍历+融合） |
| 适用数据 | 非结构化文档 | 结构化菜谱知识 |
| 强项查询 | 全局综述、跨文档主题 | 精确检索、局部推理 |
| 索引更新成本 | 高（需 LLM 重算摘要） | 低（embedding + 图插入） |
| 百万级可行性 | 社区摘要成本爆炸 | 可控（瓶颈在在线遍历延迟） |

核心判断：两条路线不是替代关系，是解决不同问题的。当前场景不需要宏观摘要能力，用在线检索换离线零 LLM 成本是符合约束的合理取舍。
