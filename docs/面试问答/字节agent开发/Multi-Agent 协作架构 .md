# FridgeAI 项目面试问答记录 — Agent 架构深度追问

> 基于 FridgeAI 智能冰箱对话系统项目的技术面试追问整理，覆盖 Agent 框架选型、子 Agent 通信、中间件、HITL、并发扩展、架构对比

---

## L1-1：你对比过哪些 Agent 框架？为什么选 LangGraph？

**主要对比了 LangGraph、CrewAI 和 AutoGen 三个方向。**

CrewAI 的优势是角色定义清晰，适合多 Agent 协作场景，但它的通信模型是对话式的，Agent 之间通过对话传递信息，对于需要精确控制工具调用和状态的场景来说，链路较长且不够透明。AutoGen 的方向类似，更偏向对话驱动的多 Agent 协商。

选择 LangGraph 的核心原因有三个：

**第一，多轮对话持久化是刚需。** 冰箱场景中用户会说"刚才那个菜呢"、"把它的做法详细说说"，这要求系统能跨轮次记住上下文。LangGraph 的 Checkpointer 机制天然支持按会话标识做对话状态的检查点存储和恢复，这是 CrewAI 和 AutoGen 原生不具备的。

**第二，需要人机审批的打断-恢复能力。** 当 Agent 要保存用户的饮食偏好（比如"我不吃辣"）时，必须让用户确认。LangGraph 的中断机制可以在工具执行前挂起，等待用户审批后通过指令原语恢复执行，这个能力是选择它的关键因素。

**第三，长期记忆的跨会话持久化。** LangGraph 的 Store 机制支持跨会话的用户偏好存储，比如用户三天后回来，系统依然知道他不吃花生。这个能力和 Checkpointer 的短期记忆形成互补。

坦白说，项目对 LangGraph 的使用是克制的——目前是单节点图（起点 → Agent → 终点），没有做复杂的状态编排。但我们利用了它底层的状态管理基础设施，这是其他框架较难提供的一体化方案。

## L1-2：3 个子 Agent 之间怎么通信？谁来做任务分发？

**三个子 Agent 之间不直接通信，采用主 Agent 集中编排的串行模式。**

架构中，主 Agent 拥有三个子 Agent 调用工具——菜谱专家、替换专家、烹饪专家。每个子 Agent 被包装成一个标准的工具函数，对外签名就是接受查询字符串、返回结果字符串的工具。主 Agent 看到用户请求后，根据系统提示词中的路由规则决定调用谁。

举个例子，用户问"冰箱里能做什么菜？缺的食材能换吗？"，主 Agent 会先调用菜谱专家获取推荐和缺失食材列表，得到结果后分析出有缺失食材，再调用替换专家查询替代方案，最后把两轮结果整合成一个连贯回复。这是一个串行链路，不是让三个子 Agent 同时工作然后投票。

**为什么不用对等通信？** 因为烹饪场景的任务有天然的依赖关系——先知道缺什么，才能问怎么替换。并行分发在这个场景下收益不大，反而增加编排复杂度。串行模式足够清晰，每一步的结果对下一步有信息增益。

**任务分发由上层的路由规则决定**，规则写在主 Agent 的系统提示词里，比如"推荐菜/搜索菜谱 → 菜谱专家"、"替换/缺 XX 怎么办 → 替换专家"、"烹饪技巧/怎么做更好吃 → 烹饪专家"。这是一种基于语义意图的静态路由，不需要额外的分类模型。

## L1-3：5 层中间件的执行顺序是什么？哪个最常被触发？

**五层中间件遵循洋葱模型，从外到内依次是：调用限流 → 摘要压缩 → 人机审批 → 模型重试 → 工具重试。**

请求从最外层进入，穿过五层到达实际执行，响应再反向穿出。各层职责：

| 中间件 | 位置 | 作用 |
|--------|------|------|
| 调用限流 | 最外层 | 硬限制 15 次模型调用，防止 Agent 陷入无限循环 |
| 摘要压缩 | 第二层 | 对话超过 4000 Token 时触发，保留最近 10 条消息，更早内容做中文摘要 |
| 人机审批 | 第三层 | 拦截偏好保存等写操作，推送前端确认卡片 |
| 模型重试 | 第四层 | 大模型 API 调用失败时，指数退避重试最多 3 次 |
| 工具重试 | 最内层 | 特定工具（检索类、子 Agent 类）失败时重试 2 次 |

**摘要压缩是最常被触发的。** 4000 Token 的阈值在多轮对话中很容易达到，每次触发还需要额外调用一次大模型做摘要，这在长对话中是常态。而模型重试和工具重试只在异常时触发，调用限流的 15 次上限在正常对话中几乎不会碰到（通常 2-8 轮），人机审批只拦截写操作，属于偶发事件。

## L2-1：你说"缩小工具选择空间"，如果不缩小会出什么问题？有没有做过对比实验？

**核心问题是工具选择的"组合爆炸"。**

在 V1 版本中，单一 Agent 直接面对全部 8 个工具——包括冰箱查询、菜谱推荐、食材替换、烹饪问答、偏好存取等。这些工具在语义上有重叠，比如"推荐菜谱"和"按食材搜索菜谱"功能相近但适用场景不同。当大模型面对 8 个语义相近的候选项时，选择准确率会下降，实际测试中就出现过用户问替换食材、Agent 却调了推荐接口的情况。

**缩小选择空间的核心思路是分层决策：** 主 Agent 只需从 3 个子 Agent 中选择（菜谱/替换/烹饪），决策空间从 8 缩小到 3。每个子 Agent 内部也只绑定 1-3 个高度内聚的工具，替换专家甚至只绑定 1 个工具，零选择成本。

**关于对比实验：** 编写了 12 条工具选择测试用例，覆盖 7 种意图类别，用工具正确性指标验证路由是否准确。测试不仅验证正向路由（问菜谱应该调菜谱专家），还验证反向约束——确保菜谱类问题不会直接调用原料层面的推荐工具、替换类问题不会绕过替换专家。从测试结果看，V3 子 Agent 架构的路由准确率相比 V1 单 Agent 有明显提升。

此外还带来两个额外收益：
- **独立温控：** 替换专家用温度参数设为 0 确保输出确定性，菜谱专家允许一定随机性保留创造力
- **独立提示词：** 每个子 Agent 的提示词短且聚焦，不会因为多角色指令挤在一起导致"指令稀释"

## L2-2：HITL 人机审批具体怎么实现？什么算"高风险操作"？审批超时怎么办？

**整个链路分四步：**

**第一步，中间件拦截。** 在 Agent 的中间件栈中，人机审批层配置了拦截规则，目前拦截的是"保存用户偏好"这个工具。当 Agent 打算调用它时，中间件在工具真正执行前将其挂起。

**第二步，前端通知。** 中断信号通过 WebSocket 推送到前端，消息包含被拦截的操作描述。前端渲染一张审批卡片，用户点击"批准"或"拒绝"。

**第三步，状态恢复。** 用户决策通过 WebSocket 回传后端，后端使用 LangGraph 的恢复指令原语，携带用户的批准/拒绝决定，恢复之前挂起的图执行。如果批准——工具正常执行，偏好写入持久化存储；如果拒绝——工具调用被跳过，Agent 继续后续流程。

**关于"高风险操作"的定义：** 目前的判断标准是是否产生持久化的数据修改。读操作（查冰箱、查菜谱）不需要审批；写操作（保存偏好）需要确认。因为 Agent 不应在用户不知情的情况下修改其过敏源、饮食禁忌等敏感信息。此外，清空冰箱、删除收藏等操作也预留了拦截点，只是当前版本尚未激活。

**关于审批超时：** 这是当前实现的一个已知局限。目前没有针对单次审批的超时清理机制，依赖的是 WebSocket 连接级别的 60 秒整体超时——如果连接断开或超时，整个流式会话终止，未完成的审批自然失效。在真实生产环境中，需要引入审批的 TTL 机制和超时后的默认策略（比如默认拒绝），这也是后续要完善的方向。

**当前局限：** 中断只在 WebSocket 流式对话路径中完整生效。如果通过 REST 接口直接调用 Agent，中断虽然触发但恢复逻辑未在 REST 接口中实现。这是为了优先覆盖主流交互路径（移动端 AI 对话界面）的取舍。

## L3-1：如果这个系统要支持 100 个并发用户同时问菜谱，你的 3 Agent 架构瓶颈在哪？怎么扩展？

**瓶颈主要在三层：**

**第一层：大模型 API 的串行调用链。** 当前一次用户请求可能触发多轮大模型调用——主 Agent 分析意图 → 调用子 Agent → 子 Agent 内部可能再调工具 → 主 Agent 整合回复。每次调用之间有依赖关系，无法并行。100 个并发用户意味着可能有 300-500 个并发的大模型 API 请求，这直接受限于 API 提供方的速率限制和 Token 配额。

**第二层：状态存储的竞争。** 当前检查点存储用了基于 SQLite 的实现，在高并发写入场景下，SQLite 的写锁会成为瓶颈——同一时间只能有一个写操作。每个用户每轮对话都要写检查点，100 个用户同时对话会产生大量写竞争。

**第三层：检索引擎的资源占用。** 每次查询涉及 Neo4j 图遍历和 Milvus 向量检索，都是独立的网络调用。100 个并发意味着检索层的 QPS 压力陡增。

**扩展方案：**

针对第一层，引入大模型调用的连接池和请求队列，用限流器平滑请求峰值；对高频重复查询（比如"查看冰箱有什么"）可以加一层语义缓存，相同意图直接返回缓存结果。

针对第二层，将检查点存储从 SQLite 升级到 Postgres 或 Redis——LangGraph 本身支持基于 Postgres 的持久化存储，能处理更高的并发写入。

针对第三层，Neo4j 和 Milvus 本身支持并发查询，关键是做好连接池管理和检索结果的缓存。另外，对于简单查询可以跳过复杂路由，直接用倒排索引命中，减少检索链路长度。

还有一个架构层面的优化方向——将 Agent 推理和检索解耦为异步流水线：用户发消息 → 检索层并行预取候选菜谱 → Agent 推理时直接消费预处理结果，减少等待时间。

## L3-2：如果把 Router + 子 Agent 换成单 Agent + 动态工具注册（MCP），设计上会有什么变化？

**核心变化体现在三个维度：**

**第一，工具发现从"编译时"变为"运行时"。** 当前架构中，三个子 Agent 的工具集合是代码中静态绑定的——菜谱专家永远只有 3 个工具。换成 MCP 后，工具通过 MCP 服务端动态注册，Agent 启动时发现可用工具列表，甚至可以在运行时热加载新的工具能力。这带来了灵活性——比如新增一个"营养分析专家"，只需要部署新的 MCP 服务端，不用改主 Agent 代码。

**第二，路由逻辑从"显式"变为"隐式"。** 当前的主 Agent 系统提示词里写着明确的路由规则——"推荐菜 → 菜谱专家"。换成单 Agent 后，路由不再是显式的条件判断，而是依赖大模型自身在众多工具中选择正确的那个。这简化了架构（不用维护路由规则），但也丢失了确定性——没法保证它一定按期望的方式选择工具。

**第三，责任边界从"清晰"变为"模糊"。** 子 Agent 架构的一个隐含优势是故障隔离——替换专家的温度参数可以设为 0 保证确定性，烹饪专家的重试策略可以单独配置。换成一个单体 Agent 后，所有工具共享同一套配置（温度、重试策略、上下文窗口），调参的自由度会降低。此外，每个子 Agent 的独立提示词也合并成一个更大的系统提示词，可能引入"指令稀释"问题。

**个人判断：** MCP 更适合工具种类多、变化频繁、需要跨系统互操作的场景（比如 IDE 场景下的代码补全 + 文件管理 + 终端操作）。对于 FridgeAI 这种工具集合稳定、领域边界清晰的垂直场景，显式路由 + 子 Agent 的方案在可控性和可调试性上更有优势。但这不意味着两者互斥——未来可以考虑混合方案，即保留子 Agent 的职责划分，但每个子 Agent 的工具通过 MCP 动态获取，兼顾灵活性和可控性。
