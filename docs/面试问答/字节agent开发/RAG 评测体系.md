## 追问 L1

### Q1: 评测数据集的问答对是怎么构建的？人工写还是有自动生成？

**回答：**

评测数据集是**人工构建**的。目前包含 50 条中文烹饪问答对，覆盖 11 个领域。

**构建方式：**

1. **覆盖烹饪场景的多样性**：先将用户可能在智能冰箱场景中提出的问题分为 11 个类别——烹饪技巧（如"如何让鸡肉更嫩"）、菜谱细节（如"番茄炒蛋需要哪些食材"）、食材知识（如"怎样挑选新鲜的鸡蛋"）、食材搭配（如"鸡蛋和番茄能做什么菜"）、菜系知识（如"川菜有什么特点"）、食材替换、新手友好、食品安全、厨房设备、膳食规划、菜谱推荐。

2. **逐条编写问题与参考答案**：每个问题手写对应的标准答案。参考答案要求包含核心知识点（而不是逐字固定答案），因为 Ragas 的 Correctness 指标是语义级评判而非字符串匹配。例如"如何去除羊肉的膻味"的参考答案写明了焯水时加白萝卜或茶叶等关键操作，而不是限制为唯一的正确措辞。

3. **标注辅助字段**：每条数据附带难度等级（easy/medium/hard）和预期实体列表。预期实体用于辅助判断检索质量——如果检索到的文档完全不包含预期实体，说明检索方向有系统性问题。

**为什么不自动生成？** 自动生成（如用 LLM 批量扩写）有两个风险：一是 LLM 生成的"参考答案"可能包含幻觉，用它来评测等于用错误标准衡量正确性；二是领域的覆盖均衡性难以保证——LLM 倾向于生成它擅长的题型，导致评测数据偏离真实用户的提问分布。

**当前局限：** 50 条样本偏少，且分布不均——烹饪技巧（13 条）和菜谱细节（11 条）占了近一半，而菜谱推荐只有 1 条。这是后续扩充的方向。另外 50 条中大部分是事实型查询，关系推理型问题较少，导致测试中路由策略高度集中在混合传统检索上。

### Q2: Faithfulness（忠实度）怎么评？用的是什么模型做 judge？

**回答：**

Faithfulness（忠实度）衡量的是**生成内容的"诚实度"——回答中的每一句话是否都能从检索到的上下文中找到依据**，有没有模型自己编造的信息。

**评判机制：** 使用 Ragas 框架的 Faithfulness 指标，底层是 LLM-as-Judge 模式。流程分为两步：

1. **声明拆解**：Judge LLM 将生成的回答逐条拆解为一组原子化的"声明"（claims），每条声明是对一个事实点的独立断言。
2. **逐条验证**：对每条声明，Judge LLM 逐一检查它是否能从检索到的上下文中被推导出来。如果能推导，声明判定为忠实；如果不能（模型中编造了上下文中不存在的信息），则判定为不忠实。最终得分 = 忠实声明数 / 总声明数。

**Judge 模型：** 使用的是 DeepSeek V4 Flash 版本（通过 EVAL_MODEL 环境变量可配置），temperature 设置为 0 以保证评分的稳定性和可复现性。

**工程适配细节：** 因为 DeepSeek API 不完全支持 OpenAI 的 response_format=json_object，在 Judge LLM 外层实现了一个 JSON 输出的包装器——在 prompt 末尾强制注入 JSON 输出指令，并内置了 JSON 修复逻辑：处理 LLM 返回空对象、多余 markdown 代码块标记、以及字符串中未转义的 ASCII 双引号等问题。兜底策略是当 JSON 彻底无法解析时，默认给 verdict=1（宽容策略，避免因解析失败而误判为不忠实）。

**阈值设定：** 当前 Faithfulness 的通过阈值设定为 0.30，这是基于实际测试结果校准的——在中文烹饪领域，50 道题的综合评测中，Faithfulness 通常在 0.30-0.60 之间。设太高会导致频繁误报，设太低失去检测意义。

## 追问 L2

### Q1: Ragas的Correctness指标依赖参考答案。你的参考答案从哪来？参考答案的质量怎么保证？

**回答：**

**参考答案来源：** 全部由本人手动撰写。每道题在编写时参考了两个维度：

1. **领域知识验证**：对于事实型问题（如"番茄炒蛋需要哪些食材""怎样挑选新鲜的鸡蛋"），参考答案基于公认的烹饪常识，确保核心知识点正确。
2. **知识库一致性**：参考答案覆盖的关键实体和信息点，需要与 Neo4j 知识图谱 + Milvus 向量库中实际存储的内容有交集。如果参考答案写了一条知识库中完全不存在的知识，Correctness 指标就永远达不到——这不是检索系统的失败，而是评测设计的失误。

**质量保证措施：**

1. **结构化标注**：每条数据除了参考答案外，还标注了 expected_entities（预期实体）。评测时可以先交叉验证——检索到的文档是否包含了预期实体。如果预期实体大量缺失，说明检索出了问题而非生成出了问题，这为指标下降提供了归因方向。

2. **Correctness 的容错机制**：Ragas 的 Correctness 指标不是做字符串精确匹配，而是 LLM 语义判断。Judge LLM 会比较生成答案和参考答案的语义覆盖度——只要生成答案涵盖参考答案中的核心知识点，即使措辞不同也算正确。这容忍了参考答案因个人知识差异产生的非关键偏差。

3. **实际局限**：50 条参考答案确实没有经过第二人交叉校验，质量依赖个人烹饪知识。对于存在争议的烹饪问题（比如"红烧肉要不要焯水"，不同流派有不同做法），参考答案反映的是主流做法而非唯一正确答案。这种情况下 Correctness 的绝对分数意义有限，更应该关注的是 Faithfulness——回答是否忠实于检索到的上下文，而非回答是否与参考答案完全一致。

### Q2: 在优化检索策略时，哪项指标最难提升？为什么？

**回答：**

**上下文召回率（ContextRecall）是最难提升的指标。** 当前其通过阈值设为 0.18，在五项指标中最低，这本身就反映了它是最薄弱的环节。

**为什么难提升：**

1. **知识库覆盖的天花板**：ContextRecall 衡量的是"参考答案中提到的信息点，检索到的上下文中覆盖了多少"。这意味着如果参考答案中提到了知识库中根本不存在的知识点，无论检索策略怎么优化，召回率都上不去。知识库只有 323 道菜谱，而烹饪领域的问题空间远大于此——比如"怎样挑选新鲜的鸡蛋"可能涉及蛋壳颜色、摇晃感、水浮测试等多个细节，知识库里不一定全部覆盖。

2. **多跳检索的碎片化问题**：有些问题的答案需要从多个文档中拼接。例如"猪肉不同部位怎么烹饪"，涉及里脊、五花肉、排骨、梅花肉等多个食材的烹饪方法，散落在不同的菜谱文档中。当前检索倾向于返回与查询最相关的 top_k 篇文档，但可能漏掉某个部位的信息，导致召回率下降。

3. **检索粒度与评测粒度的不匹配**：ContextRecall 评估的是 ground_truth 级别的信息覆盖度，但检索系统以文档/分块为单位，一个分块中可能包含大量相关信息但评测只关心其中一小部分是否覆盖了参考答案。这种"文档粒度 vs 事实粒度"的差距使得即便检索到了正确的文档，如果分块策略不够精细，也可能在 Ragas 的逐条验证中丢分。

**相对容易提升的指标对比：**

- **Faithfulness** 相对好提升，因为它只要求"生成内容不瞎编"——只要 prompt 中强调不要编造、并且温度调低，就能显著改善。
- **AnswerRelevancy** 也相对可控——让 LLM 直接回答问题而非绕弯子，是一个 prompt 工程问题。
- **ContextRecall** 则需要从数据层面（扩充知识库）、索引层面（改进分块策略）、检索层面（多路召回 + 融合排序）三个方向同时优化，是系统性的工程难题。

**当前的改进方向：** 一是扩充知识库内容（特别是烹饪技巧和食材知识类文档，当前主要靠 cooking_knowledge.json 的通用知识条目，覆盖密度不足）；二是在检索端引入查询改写，将复杂问题拆成子问题分别检索再合并，提升对多知识点问题的覆盖。
