# 面试官追问：FridgeAI 架构深度八问

> 面试日期：2026-07-22
> 项目：FridgeAI — 基于 GraphRAG 与多 Agent 的智能冰箱对话系统

---

## Q1: 你为什么选 LangGraph 而不是 AutoGen/CrewAI？

**回答：**

我选择 LangGraph 主要基于三个维度的考量。

第一，**状态管理和多轮对话的需求**。FridgeAI 是一个需要多轮对话的场景——用户可能会说"刚才第一道菜怎么做"，这就要求系统有对话状态的持久化和恢复能力。LangGraph 原生提供了基于检查点的状态图机制，我通过 InMemorySaver 实现了对话历史的自动持久化，同一个 thread_id 下的多轮请求自动继承上文。这种 StateGraph 的设计让我可以自定义状态字段，比如我把当前冰箱食材快照和用户饮食限制都放进了自定义状态中，Agent 节点每次执行时都能拿到最新的上下文。AutoGen 和 CrewAI 的核心是多 Agent 对话模式，它们在状态管理上更偏向"Agent 之间的消息传递"，而不是"图节点的状态流转"，对于需要多轮上下文继承的冰箱对话场景来说不太匹配。

第二，**生态整合成本**。我的整个技术栈是基于 LangChain 生态构建的——工具用 @tool 装饰器封装、检索模块返回 LangChain Document 对象、流式输出用 astream_events。LangGraph 与这套生态是无缝衔接的，我创建好 Agent 实例后，一行代码就能把它嵌入 StateGraph 的节点中。如果换 AutoGen 或 CrewAI，我需要做大量的适配层工作，比如把 LangChain Tool 转成它们的工具格式，把 Document 对象转成它们的消息格式，这增加了不必要的复杂度。

第三，**可控性和生产就绪度**。LangGraph 提供了中断恢复机制（支持人工审批流程），我利用这个特性实现了保存用户偏好时的确认环节。另外 LangGraph 的图结构让执行流程非常可解释——我能清楚地看到 START → recommend → END 的执行路径，未来如果要做 Prompt Chaining（分析食材→检索菜谱→排序→生成回答四步流水线），只需要在图中添加节点和边，不需要改变整体架构。AutoGen 的 GroupChat 模式在简单场景下很快，但当业务逻辑变复杂时，消息流转会变得难以追踪和调试。

简单说：LangGraph 给我的是"图执行引擎 + 状态管理 + 生态兼容"三者合一的能力，而 AutoGen/CrewAI 更适合"让多个 Agent 自由对话"的场景，两者的设计哲学不同。

## Q2: 单 Agent vs 多 Agent 的取舍是什么？

**回答：**

这个项目经历了三版迭代，正好能体现这个取舍的过程。

**V1 阶段：单 Agent + 4 个工具。** 这是最简形态，一个 Agent 持有 4 个工具，LLM 自己决定调用哪个。好处是延迟低、架构简单。但问题是当工具数量增多时，LLM 的工具选择准确率会下降——8 个工具对 DeepSeek 来说选择空间太大，偶尔会出现"该调推荐工具却调了替换工具"的路由错误。

**V2 阶段：单 Agent + 8 个工具。** 加了上下文感知和长期记忆工具后，工具总数到了 8 个，上述问题更明显了。

**V3 阶段：混合模式——1 个主 Agent（编排器）+ 3 个子 Agent（领域专家）。** 这是我的最终选择。核心思路是：复杂领域任务交给专业子 Agent 闭环处理，简单操作留给主 Agent 直接执行。

具体来说，菜谱推荐需要协调 3 个底层工具（查冰箱库存→搜菜谱→查详情），如果让主 Agent 自己协调，一次推荐可能要 3-4 轮工具调用。我把它封装成一个菜谱推荐专家子 Agent，它内部自主完成"分析需求→搜索→返回详情"的全流程，主 Agent 只需传一句自然语言。同样，食材替换和烹饪问答也各自封装为独立子 Agent。而像"查看冰箱里有什么""保存用户偏好"这种单步操作，保留为主 Agent 直属工具，不需要子 Agent 的多余开销。

**取舍的核心判断标准是：任务是否需要多步工具编排。** 如果是，就封装成子 Agent，缩小工具选择空间、提高准确率；如果不是，就直接用直属工具，避免增加延迟。这样既享受了多 Agent 的领域专注优势，又控制了架构复杂度。当前 3 个子 Agent 的粒度刚好——再多会增加延迟和维护成本，再少则路由准确率下降。

## Q3: 如果让你重做 FridgeAI，架构上会改什么？

**回答：**

有四点我会做出不同的选择。

第一，**从第一天就引入可观测性**。现在我是在 Agent 和 Graph 层面加日志，但这个粒度不够。我会在项目初期就接入 Langfuse 或类似的 LLM 可观测平台，对每一次 LLM 调用、每一个工具调用、每一轮检索做全链路追踪。这样出了问题我能精确知道是哪个环节的延迟高、哪个工具返回了空结果、哪次模型调用产生了幻觉，而不是靠翻日志猜测。

第二，**用 Prompt Chaining 替代纯 Agent 自主决策的部分关键路径**。当前我的推荐流程是 Agent 自主决定调用顺序——这在灵活性上有优势，但可控性差。比如用户问"能做什么菜"，理想流程是：分析冰箱食材特点→检索候选菜谱→按匹配度排序→生成推荐回答。这个流程是确定性的，不需要 Agent 自主决策。我在代码里已经预留了 Prompt Chaining 版本的四步流水线（分析→检索→排序→生成），但因为时间原因没有启用。如果重做，我会对推荐这条关键路径采用显式的链式编排，每个节点可以独立测试、独立优化，出问题也能精确到具体步骤。

第三，**存储层从内存方案升级为持久化方案**。当前 InMemoryStore 和 InMemorySaver 一旦服务重启，所有用户偏好和对话历史都会丢失。如果重做，我会用 PostgreSQL 替代两者——PostgresStore 做长期记忆持久化，PostgresSaver 做对话检查点持久化。同时引入 Redis 做热点数据缓存，比如频繁查询的菜谱详情和食材同义词映射。

第四，**在 RAG 模块中引入查询缓存和预热机制**。当前每次启动都要检查 Neo4j 连接、加载 Milvus 集合、初始化 BM25 检索器，冷启动时间较长。我会对高频查询（比如"西红柿炒鸡蛋怎么做"这种确定性答案）做结果缓存，减少重复的 LLM 调用和向量检索。另外会把 query analysis 这一步的 LLM 调用换成更轻量的分类模型或规则引擎——因为对于烹饪领域这种封闭场景，查询类型的分布是有限的，不需要每次都调 LLM 做路由决策。

## Q4: 你的 Agent 有没有出现过严重跑偏的情况？怎么发现的？怎么修的？

**回答：**

确实遇到过几次，我举三个典型场景。

**场景一：子 Agent 无限重试。** 在实现子 Agent 时，我最初用了 LangChain 的 ToolStrategy（即要求模型输出结构化 JSON），但在 DeepSeek 上这个功能不可靠——模型有时不会调用隐藏的 Respond 工具，导致子 Agent 收不到"任务完成"的信号，不断重试直到达到上限。我是在看日志时发现的——一次简单的菜谱推荐触发了 10+ 次模型调用，正常应该 2-3 次。修复方案是放弃 ToolStrategy，改为在 system_prompt 中用自然语言描述输出格式要求（如"使用表格：| 菜名 | 主要食材 | 难度 | 时间 |"），让模型自由输出。因为子 Agent 的输出是给主 Agent 消费的，不需要严格的结构化 Schema，自然语言格式指令已经足够。

**场景二：模型跳过工具直接编造菜谱。** 在早期测试中，我发现当用户的食材组合比较冷门（比如"苦瓜+香蕉+牛奶"）时，工具返回空结果，但模型有时候会自己编一个菜谱出来，菜名和步骤都是幻觉。我是通过对比"工具返回的原始数据"和"最终生成的回答"发现这个问题的。修复方案是多层的：第一，在 system_prompt 中反复强调"始终基于工具返回的真实数据回答，不要编造菜谱"；第二，工具在返回空结果时返回明确的 JSON 状态（如 `{"status": "no_match", "message": "暂未找到匹配的菜谱"}`），让模型明确知道这是空结果而非数据缺失；第三，加了 ModelCallLimitMiddleware（最多 15 次调用）防止模型在找不到结果时反复尝试不同的工具组合。

**场景三：DeepSeek thinking 模式导致流式输出卡死。** DeepSeek 的 thinking 模式会先输出一长段推理过程再输出正式回答，在 WebSocket 流式场景下，用户会看到几十秒的空白然后瞬间刷出一大段文字，体验很差。更严重的是，thinking 模式下 DeepSeek 不支持 tool_choice 参数，会导致工具调用行为异常。我是通过观察前端 WS 消息的时间戳分布发现的。修复方案是给子 Agent 的模型创建函数增加了 `disable_thinking` 参数，对需要精确工具调用的子 Agent 禁用 thinking 模式。

**发现的通用方法论**：我会在关键节点打印 JSON 格式的结构化日志——包括每轮 LLM 调用的耗时、工具调用的参数和返回值。跑偏的问题大部分是通过对比"预期行为"和"实际日志"发现的，而不是等用户反馈。

## Q5: 如果模型在 ReAct 循环里连续调用同一个工具 3 次，你怎么办？

**回答：**

这是一个典型的 Agent 循环失控问题，我从预防和治理两个层面来回答。

**预防层面**，我当前的架构已经有三层防护：

第一层是 ModelCallLimitMiddleware，我设置了单次对话最多 15 次模型调用，超过就强制结束。这是一个兜底机制，防止 Agent 无限循环消耗 Token 和延迟。

第二层是 ToolRetryMiddleware，它控制的是工具调用失败时的重试行为——指数退避（backoff_factor=2.0）、随机抖动（jitter=true）、最大重试 2 次、最大延迟 10 秒。它和"模型主动重复调用同一个工具"不一样——ToolRetryMiddleware 处理的是网络错误、超时等基础设施层面的失败；而模型重复调用是决策层面的问题。

第三层是工具设计层面的预防。每个工具都有明确的适用场景描述（在 @tool 装饰器的 description 中），而且工具返回的结果中包含了状态标记（如 `"status": "ok"` 或 `"status": "no_match"`），让模型能明确判断"这个工具已经完成了它的任务，不需要再调用一次"。

**治理层面**——如果这些预防手段都失效了，我会怎么做。

首先，我会在 LangGraph 的 StateGraph 层面加一个自定义中间件，记录每个工具在本次对话中被调用的次数。当检测到同一工具连续调用 3 次时，中间件在第 3 次调用前注入一条系统消息："你已连续调用该工具 3 次，前两次的结果分别是 X 和 Y，请基于已有信息给出回答，不要再次调用该工具"。这比直接拒绝调用更好——因为它给了模型一个"台阶"，让它知道可以基于已有信息做推理，而不是强迫它继续找新信息。

其次，从根本上分析为什么会出现这种循环。大概率是工具的返回结果不足以让模型做出决策——比如返回的信息太模糊、缺少关键字段、或者格式不稳定导致模型无法解析。这时候应该优化工具返回内容的清晰度和结构化程度，而不是加更多的防护层。

当前项目因为工具数量可控（6 个）、领域封闭（烹饪），实际的循环风险不高，所以在编码阶段没有实现这个检测逻辑。但如果面向生产环境，我会在 Middleware 层加入工具调用频率监控。

## Q6: 有没有模型跳过工具直接编答案的情况？你怎么防的？

**回答：**

有，而且这是 LLM 应用中最常见也最难根治的问题之一。

**出现过的具体场景**：用户问"冰箱里有什么"，模型没有调用 get_fridge_inventory 工具，直接回答"您的冰箱里有鸡蛋、西红柿和牛奶"——实际上冰箱是空的。这是因为模型在预训练阶段见过大量类似的对话模式，它倾向于"模仿"一个合理的回答，而不是真正去查数据。

**我的防护策略是分层的，从强到弱**：

**第一层：System Prompt 指令约束。** 我在 system_prompt 中加入了明确的规则："始终基于工具返回的真实数据回答，不要编造菜谱。如果工具返回空结果，如实告知用户并给出建议。"同时对于每个工具，prompt 中都有明确的使用时机指引，比如"当用户问'能做什么菜'时，直接调用 recommend_by_fridge（无需先调 get_fridge_inventory）"。这减少了模型"不知道该调哪个工具而干脆自己编"的情况。

**第二层：工具返回值的结构化设计。** 这个很关键。我的每个工具都返回 JSON 格式，并且包含明确的状态字段。比如推荐工具返回 `{"status": "ok", "recipes": [...]}` 或 `{"status": "empty", "message": "冰箱里暂时没有食材"}`。这样模型收到工具结果后，能清楚地知道"这是真实数据"还是"没有数据"。相比之下，如果工具返回的是自然语言描述，模型更容易把空结果当成"信息不完整"而自行脑补。

**第三层：子 Agent 隔离。** 在 V3 架构中，菜谱推荐、食材替换、烹饪问答这三个容易产生幻觉的领域被封装成独立的子 Agent，每个子 Agent 有自己的 system_prompt 和工具集。我在子 Agent 的 prompt 中写了更严格的反幻觉指令（如"不要编造菜谱，始终基于工具返回的真实数据"），而且子 Agent 的工具选择空间更小（1-3 个工具），降低了"选错工具"的概率。

**如果让我加强这一层**，我会引入一个"事实核查"步骤：在最终回答返回给用户之前，用一个轻量的规则检查——回答中提到的菜谱名是否存在于工具返回的 JSON 中。如果不存在，就标记为疑似幻觉并触发回退逻辑。当前因为领域封闭（只有 323 道菜谱），这个核查可以做得很快。但我也要承认，目前项目中没有做这个自动核查，主要是通过 prompt 约束和工具设计来预防。

## Q7: 如果 FridgeAI 要服务 100 万用户，你的架构瓶颈在哪？

**回答：**

坦率地说，当前架构是面向开发和演示场景设计的，要服务 100 万用户需要做较大改造。我从上到下分析瓶颈点。

**存储层——最紧急的瓶颈。** 当前对话历史和用户偏好用的是 InMemoryStore 和 InMemorySaver，数据全在进程内存中。这意味着：服务重启全部丢失；无法水平扩展（多个进程各自有独立的内存，用户 A 连到进程 1，刷新后连到进程 2，对话历史就没了）；内存上限受限于单机。解决方案是用 PostgreSQL 替换——PostgresStore 做长期记忆，PostgresSaver 做检查点持久化，两者共享同一个 PG 实例。同时引入 Redis 做会话粘性和热点缓存。

**WebSocket 连接管理——最容易被忽略的瓶颈。** 每个活跃用户维持一条 WebSocket 长连接用于流式对话。100 万用户峰值并发按 5% 估算就是 5 万条连接。单个 Python 进程的 WS 连接上限一般在 1-2 万左右。解决方案是：水平扩展 FastAPI 实例 + Nginx/HAProxy 做 WS 负载均衡 + Redis Pub/Sub 做跨实例消息广播。同时引入连接心跳和空闲断开机制。

**LLM API 调用——成本和延迟的双重瓶颈。** 当前每次用户对话都会触发多次 LLM 调用（query analysis + Agent tool-calling + generation），DeepSeek API 有速率限制和并发限制。按每天每用户 3 次对话、每次 4 次 LLM 调用计算，100 万用户每天就是 1200 万次调用。需要：引入语义缓存（相似问题直接返回缓存答案，减少重复的 LLM 调用）；对高频确定性操作（如查菜谱详情）走数据库直查而非 LLM 生成；使用异步调用 + 请求队列削峰填谷。

**Neo4j 和 Milvus——数据层的扩展。** 当前是单实例部署。Neo4j 社区版不支持集群，如果知识图谱查询成为瓶颈，需要升级到企业版或使用读写分离。Milvus 支持分布式部署（Proxy + Query Node + Data Node 分离），可以按数据量做分片。另外 BM25 检索器目前是用内存中的文档列表创建的，文档多了内存压力大，应该迁移到 Elasticsearch 做稀疏检索。

**无状态化改造。** 当前有一些模块级全局变量（如子 Agent 的单例缓存、冰箱食材的全局快照），在多进程部署时会出问题。需要把全局状态全部外部化——食材快照进 Redis，子 Agent 实例通过懒加载 + 连接池管理。

**安全和流控。** 当前没有任何认证和限流机制。100 万用户场景下必须加 API 网关做认证（JWT）、限流（令牌桶/漏桶）、和异常检测。

总结来说，瓶颈的优先级是：存储 > 安全/流控 > LLM 调用优化 > 数据层扩展 > 连接管理。

## Q8: 你的 RAG 检索延迟如果从 100ms 涨到 2s，怎么排查？

**回答：**

这是一个生产环境中很真实的故障场景。我的排查思路是按检索流水线的执行顺序逐段定位，而不是猜测。

**第一步：确认问题范围。** 先看是所有查询都变慢还是部分查询变慢。如果所有查询都慢，大概率是基础设施问题（数据库、网络、API）；如果只有特定类型的查询慢（比如只有图检索慢而传统检索正常），那就是特定模块的问题。我会先跑一轮预设的基准查询（包含简单查找、关系推理、组合查询三类），记录每类的延迟分布。

**第二步：逐段打点定位。** 我的 RAG 检索流水线是按照"查询分析 → 路由决策 → 检索执行 → 结果融合 → 生成回答"的顺序执行的，主程序中已经有计时日志（`start_time` 到 `end_time`）。我会在以下关键节点加细粒度的计时：

- **查询分析耗时**：LLM 调用 `analyze_query` 做结构化输出，这个步骤调了 DeepSeek API。如果这里慢，可能是 API 限流或网络抖动，也可能是 Pydantic 结构化输出的解析变慢。
- **向量检索耗时**：Milvus 的 `search` 操作。如果慢，检查 Milvus 的集合是否加载到内存（`load_collection` 状态）、索引类型（HNSW/IVF_FLAT 参数是否合适）、nprobe 值是否过大。
- **图检索耗时**：Neo4j 的 Cypher 查询，特别是多跳遍历和子图提取。如果慢，检查是否缺少索引（项目中有 `ensure_fulltext_indexes` 但可能在数据更新后失效）、查询计划是否走了全表扫描。
- **BM25 检索耗时**：这个通常很快（毫秒级），但如果文档数暴增或内存不足导致频繁 GC，也可能变慢。
- **重排序耗时**：Round-robin 融合 + 去重 + 分数归一化。如果检索到的文档数过多，这部分计算量会线性增长。
- **LLM 生成耗时**：生成模块调用 DeepSeek 生成最终回答。如果 prompt 中塞了太多文档导致 token 数过大，生成延迟会显著上升。

**第三步：分层排查基础设施。**
- 检查 DeepSeek API 的响应时间（API 侧限流或服务降级）
- 检查 Neo4j 和 Milvus 的服务器资源（CPU/内存/磁盘 IO）
- 检查网络延迟（服务之间的 ping 值是否正常）
- 检查是否有连接池耗尽（连接未释放导致新请求排队）

**第四步：针对性修复。** 根据定位结果：
- LLM 慢 → 给 query analysis 加结果缓存（相同或相似查询跳过分析），或换成规则引擎做路由
- Milvus 慢 → 调整索引参数（减小 nprobe），确保集合在内存中
- Neo4j 慢 → 优化 Cypher 查询，加索引，限制多跳深度
- 文档过多慢 → 在检索阶段收紧 top_k，减少进入重排序的候选数

这个排查路径的核心思想是：**先确认范围 → 再逐段定位 → 最后针对性修复**，而不是盲目调参数。
