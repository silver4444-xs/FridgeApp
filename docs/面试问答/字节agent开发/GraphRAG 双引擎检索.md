## 追问 L1

### Q1: Neo4j里你存了多少节点和关系？图谱的schema长什么样？

**回答：**

图谱规模：基于项目中的菜谱数据库，Neo4j 中存储了 323 道菜谱的完整知识图谱。图谱的 schema 包含四种节点类型和三种关系类型：

**节点类型：**

- **Recipe（菜谱）**：属性包括菜名、描述、所属菜系、分类标签、准备时间、烹饪时间、难度、份量
- **Ingredient（食材）**：属性包括名称、分类（蔬菜/蛋白质/调味品等）、营养信息、储存方式
- **Category（分类）**：如"川菜""家常菜""素菜"，用于菜谱归类
- **CookingStep（烹饪步骤）**：属性包括步骤名称、编号、烹饪方法、工具、预估时间

**关系类型：**

- `(Recipe)-[:REQUIRES]->(Ingredient)`：菜谱与食材的需求关系
- `(Recipe)-[:BELONGS_TO_CATEGORY]->(Category)`：菜谱的菜系归属
- `(Recipe)-[:CONTAINS_STEP]->(CookingStep)`：菜谱的执行步骤

这个 schema 的核心设计思路是以 Recipe 为中心，向外辐射到食材、分类和步骤三个维度，足够支撑食材替代查询、菜谱推荐和烹饪知识问答等核心业务场景。图谱数据通过 Neo4j 全文索引辅助回退搜索，分别在菜谱节点和食材节点的名称与描述字段上建立了全文索引。

### Q2: 3种路由策略的触发条件是什么？谁来做路由决策？

**回答：**

三种路由策略分别是**混合传统检索**、**图 RAG 检索**和**组合检索**。

**路由决策者：** 由智能查询路由器通过 LLM 驱动的查询分析来做决策。具体流程是：将用户查询送入 LLM，结合专用的查询分析提示词，通过结构化输出得到一个分析结果，包含查询复杂度（0-1分）、关系密集度（0-1分）、是否需要多跳推理、实体数量等维度，最终输出推荐策略及置信度。

**触发条件：**

- **混合传统检索**：适用于简单信息查找场景，比如"红烧肉怎么做""鸡胸肉有什么营养"。查询复杂度低（0-0.3），主要是关键词匹配和单实体查询，走 BM25 + 向量检索 + 图索引的三路融合路线。
- **图 RAG 检索**：适用于需要多跳关系推理的复杂场景，比如"为什么川菜用花椒而不是胡椒""哪些食材和番茄营养互补"。查询复杂度高（0.8-1.0），或者关系密集度在 0.8 以上，需要遍历图谱中的多跳路径来做结构化推理。
- **组合检索**：适用于中等复杂度的查询，将 top_k 均分给两种引擎，分别检索后做轮询交替融合，取两者的并集去重。

此外，还有一个**基于规则的降级机制**：当 LLM 调用失败时，通过中文关键词判断（如"为什么""如何""关系""影响"）来确定复杂度，保证系统的鲁棒性。

**实际测试结果：** 在 50 条烹饪问答测试中，由于当前测试数据集偏重事实型查询，混合传统检索占据了大多数，这也是后续需要补充更多关系推理型测试用例来提升路由多样性的方向。

### Q3: 为什么选Milvus而不是其他向量数据库？

**回答：**

选型考量主要有以下几点：

**1. 性能与索引能力：** Milvus 的 HNSW 索引在百万级向量的近似最近邻搜索上表现出色，支持 GPU 加速索引构建，这对未来数据规模扩展很重要。配置了 M=16、efConstruction=200 的索引参数，在召回率和构建速度之间取得了好的平衡。

**2. 多模态字段支持：** Milvus 的集合模式支持丰富的标量字段过滤（定义了 12 个字段，包括节点类型、分类、菜系、难度等元数据），可以在向量相似度搜索的同时进行标量过滤，实现"在川菜中找和我冰箱食材最匹配的菜谱"这类混合查询。

**3. 生产级可靠性：** 支持分布式部署、数据持久化、动态加载集合，相比 FAISS 等纯内存方案更适合需要持续运行的在线服务场景。线上服务可以动态加载已有集合，不需要每次启动重建索引。

**4. 中文嵌入生态兼容：** 使用 BAAI/bge-small-zh-v1.5 中文嵌入模型，输出 512 维向量，Milvus 的余弦相似度度量与归一化嵌入直接兼容。

对于这个小规模项目，FAISS 或 Chroma 也能满足需求，但选择 Milvus 更多是考虑架构的可扩展性和未来向生产环境迁移的平滑性。

## 追问 L2

### Q1: 实体级和主题级检索有什么区别？各适用于什么场景？Round-robin融合的具体做法是什么？

**回答：**

**实体级检索**针对的是**具体的、可命名的实体**——食材名称（鸡胸肉、番茄）、菜谱名称（红烧肉、麻婆豆腐）、工具名称（平底锅）。它的检索路径是：先在图索引的内存 KV 存储中做精确匹配，如果没有命中则通过同义词扩展（比如"番茄"扩展到"西红柿"），再通过子串匹配和模糊匹配逐级降级。如果图索引结果不足，回退到 Neo4j 的全文索引做关键词搜索。适用场景："番茄炒蛋怎么做""鸡肉配什么食材"——查询中有明确的实体锚点。

**主题级检索**针对的是**抽象概念和烹饪主题**——饮食风格（川菜、减肥餐）、烹饪技法（爆炒、清蒸）、营养诉求（低热量、高蛋白）。它的检索路径不同：从图索引的关系级键值对中查找，同时利用分类信息进行匹配（比如匹配到"川菜"分类节点后，拉取所有属于该分类的菜谱）。适用场景："减肥期间吃什么""川菜有什么特点"——查询中不指定具体食材，而是围绕一个主题展开。

**关键区别总结：** 实体级检索以实体名为锚点，走的是"点→邻居"的图遍历思路；主题级检索以分类和关系为锚点，走的是"面→节点"的语义扩散思路。

**Round-robin 融合的具体做法：**

在组合检索策略中，当两种引擎都返回结果时，不是简单地把两个列表拼起来按分数排序——那样会导致某个引擎的结果占据头部位置，另一个引擎的高质量结果被挤到后面。

具体做法是：
1. 将 top_k 均分，比如 top_k=10，则两种引擎各取 5 条
2. 交替排列：先放一条图 RAG 结果，再放一条传统检索结果，如此轮询交替
3. 对文档内容的前 100 个字符做哈希去重，避免两种引擎返回同一份文档
4. 截断到 top_k 条返回

这样做的目的是保证结果列表的**多样性**——用户既能看到基于结构化关系推理得出的结果，也能看到基于语义相似度匹配得出的结果。

在混合传统检索内部，还有一套三路融合的打分机制：BM25 稀疏检索权重 0.3 + 向量检索权重 0.5 + 图索引检索权重 0.2，并针对烹饪知识类文档给 1.4 倍的分数补偿。

### Q2: 如果图检索返回空结果但向量检索有结果（或反过来），你怎么处理？

**回答：**

这是一个重要的容错问题。项目中有两层兜底机制：

**第一层：检索引擎内部的回退。** 在混合检索模块中，如果实体级检索的图索引没有命中，会自动回退到 Neo4j 全文索引做关键词搜索；如果主题级检索的图索引也没有命中，则回退到 Neo4j 的 CONTAINS 查询（对分类、菜系、标签字段做模糊匹配）。向量检索同理，如果 Milvus 的过滤搜索没结果，会去掉过滤条件重新搜索。

**第二层：查询路由器层面的降级。** 如果图 RAG 检索完全失败（抛异常或返回空结果），路由器会捕获异常并降级到混合传统检索，确保用户至少能拿到语义匹配的结果。反过来，如果传统检索返回空但图 RAG 有结果，则直接使用图 RAG 的结果。如果两个都失败，会返回一个明确的"未找到相关信息"提示。

实际场景中，图检索空结果通常发生在查询中包含图谱中不存在的实体时（比如用户问了一个小众食材），这时向量检索的优势就体现出来了——即使精确匹配不到，语义相似度也能找到相近的内容。反过来，向量检索可能在处理需要精确关系推导的查询时力不从心（比如"A 食材和 B 食材在营养上有什么关系"），这时图检索的结构化遍历能力是向量检索无法替代的。

### Q3: Milvus的索引类型选了什么？为什么？

**回答：**

选了 **HNSW**（分层可导航小世界图）索引，度量类型为**余弦相似度**。

**为什么选 HNSW：**

1. **搜索性能：** HNSW 在近似最近邻搜索中是目前综合表现最好的算法之一，查询时间复杂度接近 O(log N)，在 323 道菜谱的知识库规模下实测延迟极低。它的分层跳表结构允许搜索时从高层稀疏图快速逼近目标区域，再在底层稠密图精细搜索。

2. **召回率与速度的平衡：** 通过 M=16（每层最大连接数）和 efConstruction=200（构建时动态候选列表大小）的参数配置，在索引构建速度和最终召回率之间取得了好的平衡。搜索时 ef=64 的配置也保证了足够的候选集质量。

3. **内存开销可控：** 相比 IVF 类索引，HNSW 不需要额外的聚类中心存储，内存占用主要来自图结构本身。向量维度是 512，属于中等维度，HNSW 在这个维度范围内表现最优。

**为什么选余弦相似度：** 嵌入模型 BGE-small-zh-v1.5 输出的向量经过了归一化处理，归一化后余弦相似度与内积等价。余弦相似度对向量的绝对长度不敏感，只关注方向一致性，这在文本语义相似度任务中比欧氏距离更合适。

## 追问 L3

### Q1: 如果知识图谱规模扩大10倍（百万级节点），Neo4j查询性能会怎么变化？你会怎么优化？

**回答：**

**性能变化预测：**

当前系统的核心瓶颈会在几个地方暴露：

1. **多跳遍历查询**：目前使用的可变长度路径查询 `[*1..max_depth]`，在百万级节点上如果不加限制，路径组合会呈指数爆炸。例如深度为 3 的遍历，在一个平均度为 10 的图中，路径数可达 10³=1000 条；如果平均度增加到 50，就是 125000 条。延迟可能从毫秒级变成秒级。

2. **全文索引回退查询**：Neo4j 的全文索引基于 Lucene，在百万级文本量下性能尚可，但如果回退频率高且并发量大，会成为瓶颈。

3. **图索引构建**：目前系统启动时从 Neo4j 全量拉取数据构建内存中的 KV 索引，百万级节点意味着几十 GB 的内存占用，当前的纯内存方案不可持续。

**优化方案：**

1. **查询层面**：首先给所有遍历查询加上硬性的深度限制和节点数上限。对于高频查询模式（比如"某个菜谱需要什么食材""某个分类下有哪些菜谱"），可以预计算并缓存结果到 Redis，避免实时遍历。使用 Neo4j 内置的 `shortestPath()` 替代自写的路径遍历函数，因为它是经过优化的内置算法。

2. **索引层面**：为度数高的"超级节点"（如热门食材"盐""酱油"）建立属性索引和约束，避免全表扫描。Neo4j 5.x 支持复合索引和关系索引，可以针对 `REQUIRES` 等高频关系类型创建索引。

3. **架构层面**：引入读写分离——图索引构建从只读副本拉取数据，不影响在线查询的写入操作。内存 KV 存储改为基于 RocksDB 等嵌入式存储的磁盘方案，或者使用 Redis 做分布式缓存。对于大规模图上的复杂分析查询（如社区发现、相似度聚类），可以在 GNN 框架（如 PyG、DGL）中离线训练嵌入，将图结构信息蒸馏为向量存入 Milvus，在线查询时直接用向量相似度替代部分图遍历。

4. **分库分图**：按菜系或食材大类将大图拆分为多个子图，查询时先通过路由定位到目标子图，减少单次查询的搜索空间。

### Q2: 如果要用LightRAG或GraphRAG最新方案重做，你的架构会怎么调整？

**回答：**

**背景理解：** LightRAG 的核心创新在于引入了社区摘要（Community Summary）机制——从知识图谱中自动发现语义社区，为每个社区生成一个高层级的摘要文本用于检索。相比目前的"先查图再喂 LLM"模式，LightRAG 把图的结构信息提前转化为了文本摘要，检索时直接匹配合适的社区摘要，减少了在线图遍历的延迟。微软的 GraphRAG 也采用类似思路，使用 Leiden 算法做层次化社区发现。

**调整思路——不是推倒重来，而是渐进增强：**

**1. 保留现有的双引擎架构作为基础。** Neo4j + Milvus 的组合已经被验证在精确实体查询和语义检索上互补性很好，这是项目的核心资产，不应该被替换掉。LightRAG 适合作为一个**新增的检索通道**，而不是替代品。

**2. 引入离线社区摘要管道：**
- 用 Leiden 或 Louvain 算法对 Neo4j 中的烹饪知识图谱做社区发现，预期会自然涌现出"川菜社区""烘焙社区""减脂餐社区"等语义聚类
- 为每个社区（如包含 20-50 个相关菜谱节点及其食材/步骤关系）用 LLM 生成一个社区摘要——这是一个"宏观视角"的文本描述，比如"川菜社区以麻辣为核心，代表性食材是花椒和干辣椒，常见技法为爆炒、水煮、干煸"
- 将这些社区摘要向量化存入 Milvus 的新集合中

**3. 路由策略扩展为四路：** 在现有的三策略基础上增加"社区摘要路由"。当用户的查询带有明显的宏观主题特征（如"我想了解川菜的烹饪哲学""意大利菜和中国菜在食材使用上有什么不同"），优先走社区摘要检索——返回的不是单个菜谱，而是整个菜系或主题的全景图。

**4. 在线流程优化：** 将当前的"图遍历→结果→LLM 推理"流程替换为"社区摘要检索→补充实体检索→LLM 生成"。"先给宏观视角再补充细节"的模式更贴近人类回答问题的方式，也能减少在线图查询的压力。

**5. 增量更新机制：** LightRAG 的一个重要特性是支持增量添加文档而不需要全量重建社区，这对于菜谱持续扩展的场景很有价值——新增一道菜谱时，只需更新它所属社区的摘要和邻接关系，而不用重新跑全图聚类。

**6. 不打乱现有技术栈：** 社区发现用 Python 的 networkx + community 库即可，社区摘要生成复用现有 DeepSeek API，社区摘要向量入 Milvus 新集合。所有改动控制在新增模块范围内，不影响现有三个检索引擎的正常工作。

这样调整后，系统从"查图→搜文档→回答"两级架构升级为"社区摘要（宏观）→ 结构化关系（中观）→ 实体检索（微观）"三级架构，每一层对应不同粒度的用户意图，路由策略的选择空间更大，回答的层次感也会更好。
