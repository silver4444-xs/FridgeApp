## 追问 L1

### Q1: 60+组同义词怎么构建的？手动标注还是有自动化流程？

**回答：**

目前是**手动标注**的方式。在模糊匹配模块中维护了一个包含约 60 组、120+ 条映射条目的同义词字典，覆盖常见食材的别名、方言称呼和简写。

构建过程分为几个来源：

1. **日常等价关系**：如番茄↔西红柿、土豆↔马铃薯↔洋芋、鸡蛋↔蛋↔鸡子。这些是最常见的同物异名，来自日常生活经验。
2. **烹饪领域的细分变体**：如酱油↔生抽↔老抽（烹饪中常互相替代）、醋↔白醋↔陈醋↔香醋、淀粉↔生粉↔玉米淀粉↔太白粉。这些考虑了烹饪场景中食材的可替代性，而不仅是严格语义等价。
3. **口语简称**：如鸡胸肉↔鸡胸↔鸡脯肉、排骨↔猪排骨↔肋排。

**设计原则：** 字典保持双向对称性——每个 A→B 映射都必须有对应的 B→A 映射。项目中通过单元测试自动化校验了对称性和无自引用约束，确保字典质量。

**为什么没有自动化流程？** 烹饪领域的同义词关系带有场景依赖（比如"酱油"和"生抽"在某些场景下可互换，在严格场合不可以），纯基于词向量的相似度匹配无法区分这种细微差别。自动化的可行方向是用 LLM 辅助发现候选同义词对，再由人工确认，但目前数据集只有 323 道菜谱，涉及的核心食材不超过 200 种，手动维护成本完全可控。

### Q2: 为什么用BM25而不是TF-IDF或其他稀疏检索方法？

**回答：**

BM25 相对于传统 TF-IDF 有两个关键改进，在我们的混合检索场景中直接体现出了优势：

**1. 词频饱和机制：** TF-IDF 的评分随词频线性增长，如果一个菜谱文档中"番茄"出现了 20 次，TF-IDF 会给它 20 倍的权重，这会严重偏向长文档或重复提及某个食材的文档。BM25 引入了词频饱和曲线——词频超过一定阈值后，额外出现的边际贡献趋近于零。烹饪场景中，一道番茄炒蛋的菜谱可能在描述、步骤、小贴士中多次提到"番茄"，BM25 不会因此给它过高的分数。

**2. 文档长度归一化：** BM25 通过参数 b 控制文档长度的影响。在我们的菜谱数据中，文档长短差异很大——一道简单家常菜（如番茄炒蛋）可能只有 200 字，而一道复杂宴席菜可能有 800 字。TF-IDF 天然偏向短文档（因为短文档中每个词的密度更高），而 BM25 可以通过调整 b 参数来平衡，默认的 b=0.75 意味着长文档会受到一定的长度惩罚，但不会像 TF-IDF 那样被过度压制。

**3. 工程考量：** 使用 LangChain 社区版的 BM25Retriever，它基于 rank_bm25 库，可以直接以 LangChain Document 对象为输入，与我们已有的文档管道无缝集成，不需要额外的分词适配层。

在我们的三路融合架构中，BM25 负责稀疏检索通道，权重 0.3，与向量检索（权重 0.5）和图索引检索（权重 0.2）互补。BM25 擅长精确关键词匹配（比如直接搜"红烧肉"），而向量检索擅长语义泛化（比如搜"口感软烂的肉类菜"），图索引擅长结构化关系（比如搜"使用花椒的川菜"）。

### Q3: 三层覆盖各自做了什么？为什么需要三层而不是只在搜索时做一次替换？

**回答：**

三层覆盖如下：

**第一层——索引构建时展开：** 在构建倒排索引阶段，对于每道菜谱的每种食材，不仅以食材的原名建索引，还同步展开所有同义词作为索引键。例如，"番茄炒蛋"这道菜不仅在"番茄"键下有索引，在"西红柿"键下也有。

**第二层——搜索输入时展开：** 当用户的冰箱食材列表传入时（比如冰箱里有"有机西红柿 6 个"），先经过归一化清洗（去掉"有机""6个"变成"西红柿"），然后展开同义词加入"番茄"，使得搜索集合同时包含"西红柿"和"番茄"。

**第三层——匹配校验时展开：** 在模糊匹配器的最终判定环节，对候选食材再次做同义词展开后与冰箱食材集合求交集，同时额外加入子串匹配兜底。

**为什么需要三层而不是只做一次？**

因为三层面对的数据形态不同：

- 只做第一层（索引端展开）：如果用户的冰箱输入中包含"马铃薯"（没有被展开），而索引中只有"土豆"，就无法命中。实际上我们确实在字典里维护了"洋芋→土豆, 马铃薯"的映射，但用户可能输入任何未覆盖的别名。
- 只做第二层（搜索端展开）：如果菜谱数据中使用了用户冰箱里没有的同义词变体（比如某道菜使用了食材别名"青花菜"而非"西兰花"），搜索集合无法覆盖。
- 只做第三层（匹配端展开）：会丢失倒排索引 O(1) 的直接查找优势——前两层的展开使得大多数查询可以直接命中索引键，不需要走到第三层的遍历匹配。

**三层叠加的实际效果是：** 无论同义词的缺失发生在菜谱数据侧还是用户输入侧，都能被其中一层兜底。同时前两层保证了绝大多数查询的 O(1) 索引定位性能，第三层作为安全网处理长尾情况。这本质上是一种**纵深防御**的设计——不在单点上依赖完美的数据覆盖，而是让每一层消除一类特定的匹配失败风险。

## 追问 L2

### Q1: jieba分词在烹饪领域有专有名词问题吗？你怎么处理的？

**回答：**

坦白说，当前项目的关键词提取主要依赖 **LLM 驱动的语义提取**而非 jieba 分词。在混合检索模块中，关键词提取的实际流程是：

1. **主路径——LLM 结构化提取：** 通过专用提示词调用 LLM，输出结构化的实体级关键词和主题级关键词。LLM 天然理解上下文语义，对于"拔丝地瓜"会自动识别为一个完整实体（菜品名）而非"拔丝+地瓜"，对于"宫保鸡丁""鱼香肉丝"等专有名词也能正确识别。
2. **降级路径——正则提取：** 当 LLM 调用失败时，使用正则表达式 `[一-鿿]{2,}` 提取连续汉字作为关键词。这种方式对"拔丝地瓜"也能正确提取（作为连续 4 个汉字），但会丢失语义边界。
3. **最终兜底——字符级二元组：** 极端情况下用二元组分词来保证系统不中断。

**关于 jieba 在烹饪领域的挑战：** 如果使用 jieba 默认分词，确实会遇到专有名词切分问题。比如"拔丝地瓜"可能被切成"拔丝/地瓜"，"麻婆豆腐"切成"麻婆/豆腐"，"夫妻肺片"切成"夫妻/肺片"。解决方案通常有两种：

- **自定义词典**：将烹饪领域的专有名词（菜品名、特殊食材、技法术语）添加到 jieba 的用户词典中，调整词频使其优先成词。
- **LLM 辅助分词**：使用 LLM 一次性完成分词和关键词提取，这是当前项目实际采用的方案。

当前选择 LLM 路线的原因是项目规模下 LLM 延迟可接受（DeepSeek API 调用约 1-2 秒），且 LLM 在理解烹饪语义、区分实体关键词和主题关键词方面远优于纯统计分词。但若未来需要降低 LLM 调用成本，会考虑 jieba + 自定义烹饪词典的混合方案。

### Q2: 同义词映射有没有引入噪音？比如"土豆"和"马铃薯"替换后，结果里有不相关的"土豆泥"？

**回答：**

**不会引入这类噪音。** 原因是同义词映射作用在**归一化的食材名称**层面，而非任意文本的字符串替换。

整个匹配流程的关键环节：

1. **倒排索引的键是归一化的食材名**（如"土豆""番茄"），不是菜谱文档的全文。索引构建时，每道菜谱的每种食材被归一化后作为索引键，同义词展开只影响索引键的覆盖范围，不会修改菜谱文档本身。

2. **匹配流程（以 FuzzyMatcher.is_match 为例）** 是：将菜谱的食材列表归一化 → 展开同义词得到候选集合 → 与冰箱食材集合求交集。整个过程操作的是离散的食材名称字符串集合，而非全文搜索替换。例如"土豆泥"不是一种食材名称，它不会出现在菜谱的食材列表中（食材列表里写的是"土豆"），所以不会被展开或错误匹配。

3. **BM25 和向量检索通道**直接对原始查询和文档文本进行操作，不受同义词映射影响。同义词仅作用于图索引检索和倒排索引检索的实体匹配环节。

**但确实存在一种潜在的边界问题：** 同义词映射中的某些映射是"近似等价"而非"严格等价"。比如"酱油/生抽/老抽"三者，在烹饪实践中并非总能互相替换——老抽主要上色、生抽主要提鲜。如果用户精确想找使用"生抽"菜谱，同义词展开后会把只使用"老抽"的菜谱也匹配进来。对此的缓解措施是：同义词展开主要保障召回（不错过可能相关的菜谱），后续由多路融合的排序和 LLM 重排序来保证精准度（把最相关的排在前面）。

## 追问 L3

### Q1: 323道菜谱是静态数据。如果要支持每天更新100道新菜谱，你的索引更新策略是什么？

**回答：**

当前系统是启动时全量构建索引的模式，这对静态数据没问题，但对日增 100 条的频率就需要改为增量更新策略。我会分几个层面来设计：

**1. 倒排索引层（增量友好，改动最小）：** 倒排索引本身就是基于字典的追加结构，天然支持增量。新增菜谱时，提取其食材列表，归一化并展开同义词，直接追加到对应的索引键。删除菜谱时从对应键的集合中移除菜谱 ID。这层不需要重构。

**2. 向量索引层（批量增量 + 定期重建）：** Milvus 支持动态插入数据，新增菜谱分块后生成向量直接插入集合。但 HNSW 索引在大量增量插入后会逐渐退化——新点的连接质量不如全量重建。策略是：日常执行增量插入（每天 100 条对 HNSW 的退化影响很小），每周或每千条新数据后触发一次后台全量索引重建。得益于 Milvus 支持动态加载集合，重建期间在线服务可以继续使用旧索引，重建完成后一键切换，对用户无感知。

**3. 图索引层（增量最复杂）：** 目前的内存 KV 存储启动时从 Neo4j 全量加载。改为增量更新需要：

- 新菜谱先写入 Neo4j 图（创建 Recipe 节点 + 关联 Ingredient/Category/Step），这一步是实时的
- 图索引 KV 存储支持热更新：新增实体键值对和关系键值对直接追加，去重逻辑改为对新旧数据一起去重，而不是等启动时全量执行
- 为避免内存无限增长，设置基于 LRU 的淘汰策略，不常用实体可卸载到磁盘（如 RocksDB）

**4. 更新频率策略：**

- **准实时（分钟级）**：倒排索引更新、Neo4j 图节点新增
- **小时级批量**：Milvus 增量插入（攒够一批再插入，减少碎片）
- **周末/低峰期**：Milvus 索引重建、图索引全量去重优化

**5. 一致性保障：** 对于更新期间的查询，新菜谱在倒排索引和 Neo4j 中立即可见（秒级），在向量索引中可见有小时级延迟（取决于批量窗口）。这在烹饪场景中是可接受的——毕竟用户不会因为一条新菜谱没被向量检索覆盖而有明显体感差异，大多数查询走的是倒排索引和图索引。

**6. 数据源对接：** 如果每天 100 条来自外部爬虫或编辑后台，可以引入一个消息队列（如 Redis Stream），新增菜谱事件推送到队列，各索引层作为消费者独立消费——某个索引更新失败不影响其他层的正常服务，也便于后期横向扩展各索引服务的实例。
