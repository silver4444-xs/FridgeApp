# 面试官追问（第二轮）：架构扩展与检索冲突

> 面试日期：2026-07-22
> 项目：FridgeAI — 基于 GraphRAG 与多 Agent 的智能冰箱对话系统

---

## Q1: 你的 3 个子 Agent 之间有没有出现过路由错误？比如应该调菜谱 Agent 却调了替换 Agent？你怎么处理的？

**回答：**

确实遇到过，而且这个问题在 V2 到 V3 的架构迭代中是一个核心关注点。

**出现过的情况**：用户说"我没有黄油，能做什么甜点？"——这句话包含两个意图：缺少食材（黄油到替换专家）和菜谱推荐（做什么甜点到菜谱专家）。DeepSeek 有时会只路由到替换专家，给出黄油的替代方案后就结束了，忽略了用户真正的需求是做甜点推荐。

**我处理的方式分三个层面**：

第一，**System Prompt 路由规则细化**。在 V3 的主 Agent system prompt 中，我明确写了路由规则——比如"凡是涉及「推荐菜/做什么菜/搜索菜谱/菜的做法」调用 recipe_expert，凡是涉及「代替/替换/缺XX怎么办」调用 substitution_expert"。这些规则包含了具体的中文关键词触发条件，降低了模型在路由决策上的模糊空间。

第二，**工具描述中的示例引导**。三个子 Agent 各自被封装为独立的 @tool，工具描述中包含了中文引导示例。比如 recipe_expert 的描述是"菜谱推荐专家：根据食材推荐可制作的菜谱。适用于「能做什么菜」「推荐几个菜」「XX菜怎么做」"。这样当模型读到用户的自然语言时，描述中的示例能与用户意图做语义对齐。

第三，**允许多专家协同调用**。我没有把架构设计成"一个问题只调一个专家"。主 Agent 可以看到所有 6 个工具（3 个直属 + 3 个子 Agent），它可以在同一轮对话中先调 recipe_expert 获取菜谱，再调 substitution_expert 查替换——这正是 LangChain create_agent 的 tool-calling 范式支持的。对于"我没黄油能做什么甜点"这种复合意图问题，正确的行为就是两个专家都调。

**实事求是地说**，当前的路由决策完全依赖 LLM 的语义理解（system prompt + 工具描述），没有额外的规则引擎做兜底。在 323 道菜谱的封闭烹饪领域中，意图范围有限，LLM 路由准确率足够。但如果面向更开放或术语密集的场景，我会加一层轻量级文本分类模型做预路由。

## Q2: 如果要把 3 个子 Agent 开放给外部开发者贡献新的 Agent（比如"烘焙专家"），你的架构要怎么改？

**回答：**

这是从一个单体项目到插件化平台的设计问题，我会分三步改。

**第一步：定义 Agent 注册规范**。当前子 Agent 是硬编码在代码中的，改造的第一步是抽象出一个 Agent 描述协议。插件开发者只需提供三个要素：身份定义（名称、自然语言描述、适用场景关键词）、能力声明（持有的工具列表及描述）、行为约束（system prompt、温度参数、是否需要禁用 thinking 模式）。这样烘焙专家的开发者不需要了解 LangChain 的 create_agent API 细节，框架自动完成实例化。

**第二步：插件加载机制**。把子 Agent 工具列表从硬编码变成可发现、可热加载的注册表。约定一个插件目录，每个插件是一个 Python 包，包含 manifest 描述文件和 agent 实现。启动时扫描目录、验证声明格式、自动注册到主 Agent 的工具列表。

**第三步：路由策略升级**。当前 3 个子 Agent 的路由靠 LLM 看 system prompt 和工具描述，在 3 个专家时够用。但如果扩展到十几个，LLM 的 tool-calling 选择空间会显著增大，路由准确率下降。我会引入两层路由：第一层是轻量级意图分类器，将用户问题映射到"推荐/替换/知识/烘焙"等大类，只暴露对应大类的工具给 LLM；第二层才是 LLM 精挑。

**工程化问题**：工具冲突检测（两个专家都声明能处理同一问题时需要声明优先级）、沙箱隔离（外部插件只能调公开 API，不能直接读写数据库）、版本管理（每个插件有独立的版本号和兼容性声明）、可观测性（自动采集每个插件的调用次数、成功率、延迟）。

## Q3: GraphRAG 的图检索和向量检索结果不一致时你怎么处理？比如图检索返回"A食材可以替代B"，向量检索返回"不能替代"。

**回答：**

这是一个触及混合检索系统核心矛盾的问题。

**当前系统的融合机制**：当选择 Combined 组合策略时，查询路由器同时调用传统混合检索（BM25 + 向量）和图 RAG 检索（Neo4j 图遍历），然后使用 Round-robin 轮询方式交替合并结果——图 RAG 的结果优先排在前面（因为图结构检索通常提供更丰富的关系信息，质量更高）。但这种合并本质上是"加和"而非"裁决"——两种结果都会被呈现给 LLM，由 LLM 在生成阶段自己判断。

**Round-robin 融合的局限性**：当前项目没有做冲突检测和置信度加权。两个矛盾的结论都会被保留并送入 LLM 生成模块，依赖 LLM 的推理能力来判断哪个更可信。这在大多数情况下有效（LLM 有一定的常识推理能力），但从系统设计角度不够严谨。

**如果重新设计冲突解决机制**：

引入**信号可信度模型**。给每个检索来源分配动态信任度——对于关系推理类查询，图检索信任度更高（图结构天然适合表达实体间关系）；对于语义相似度查询，向量检索信任度更高（Embedding 擅长捕捉语义层面的相似）。信任度的计算基于历史数据——统计每种查询类型下各检索源的答案最终被 RAGAS 评测判定为正确的概率。

实现**冲突显式化 + LLM 裁决**。当检测到同一主题有矛盾结论时，不强行选一个，而是把矛盾信息显式标注后交给 LLM："图检索基于知识图谱结构认为 A 可替代 B，但向量检索基于语义相似度未找到支持证据。请基于两者的推断路径和置信度给出判断并说明理由。"

**根本解法：数据层面减少冲突**。图检索和向量检索出现矛盾，本质上说明知识图谱的结构化关系和向量空间的语义分布不一致。如果是高频冲突，我会检查 Neo4j 中关系标注是否正确，或者 Embedding 模型对烹饪领域食材的语义区分度是否足够。

## Q4: 如果不用 Neo4j，有没有更轻量的方案？比如全用向量检索 + 结构化 prompt 引导？

**回答：**

完全可以。对于 FridgeAI 323 道菜谱的规模，全量向量检索加结构化 prompt 引导是值得认真考虑的轻量化方案。但需要想清楚"失去图结构"意味着什么，以及如何弥补。

**Neo4j 在 FridgeAI 中实际承担的角色**：存储"食材到菜谱到步骤到营养"的多跳关系网络。核心价值不是存储数据，而是支持多跳关系遍历和子图推理。比如用户问"鸡蛋、西红柿、面粉能做什么"，传统检索只能找同时包含这三个食材的菜谱；而图检索可以走"鸡蛋到西红柿到西红柿炒鸡蛋到类似菜谱到番茄蛋汤到面粉到鸡蛋饼"的多跳路径，发现跨菜谱的关联推理——这是纯向量检索做不到的。

**三种轻量替代方案的对比**：

**方案 A：全向量 + 结构化 prompt 模板**。把菜谱拆成结构化 chunk（食材段、步骤段、口味段、营养段），分别做向量嵌入。不用 Neo4j 做多跳遍历，而是在 prompt 中引导 LLM："以下是匹配到的最相关 10 道菜谱的完整信息，请分析它们之间的食材重叠关系、口味相似度，推断用户的偏好模式"。本质上是把"图推理"从检索阶段延迟到生成阶段，让 LLM 代替 Neo4j 做关系推理。代价是对 LLM 推理能力要求更高，精确多跳场景可能出错或遗漏。

**方案 B：Milvus + SQLite 混合（我会选这个）**。保留 Milvus 做语义检索，把图结构数据存入 SQLite，用递归 CTE 或应用层 BFS 模拟多跳遍历。323 道菜谱规模下 SQLite 完全能承载。相比 Neo4j，SQLite 零部署、零运维、可嵌入，启动时间从秒级降到毫秒级。代价是复杂图查询需要手写 SQL，不如 Cypher 直观——但 FridgeAI 的图查询模式相对固定，可以用预定义 SQL 模板实现。

**方案 C：用 Milvus 元数据模拟图结构**。把图关系作为元数据存入 Milvus 向量条目中。检索时先用语义搜索找候选集，再用元数据过滤做图扩展——本质上还是两次向量检索，但通过元数据模拟了一跳图遍历。

**我的判断**：如果未来知识图谱扩展到上万个节点（加入营养学知识、中医食疗、食物相克等），SQLite 的图查询性能和复杂关系表达能力会成为瓶颈，那时候 Neo4j 就是必要的了。选择什么方案取决于数据规模和查询复杂度——323 个菜谱节点用图数据库确实有点"大炮打蚊子"，但架构上预留了扩展空间也不是坏事。
